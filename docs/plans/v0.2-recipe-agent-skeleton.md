# v0.2 — Recipe Agent Skeleton

**Status:** planning draft. Architecture and rationale live in the design doc
[`docs/design/recipe-agent.md`](../design/recipe-agent.md); this is the v0.2 work
breakdown. Runtime-wide concepts (AgentOS, Postgres + pgvector) are defined in
[`Agents.md`](../../Agents.md).

**Goal:** Build the recipe agent's **code skeleton** — nothing more. v0.2 adds the
`recipe_agent/` module layout and *typed stub definitions* (the `Recipe` model, the
`RecipeIngester` interface, a `HowToCookIngester` stub, a `RecipeSink` write
endpoint, and a thin recipe-agent module), and registers a placeholder agent into
an
[AgentOS](https://docs.agno.com/agent-os/introduction) instance so the app still
boots.

Like v0.1 ("engineering-complete but business-empty"), v0.2 is
**"structurally-complete but behavior-empty"**: the modules exist, are typed,
import cleanly, and the app boots — but nothing is parsed, embedded, retrieved, or
answered. Database wiring, ingestion, RAG, real agent behavior, and **all user
profile / personalization work** are deferred (see Out of Scope).

**Definition of Done:**

- `src/spectres_runtime/recipe_agent/` exists with the §1 layout and imports
  cleanly.
- `Recipe` is a typed model with **structured `ingredients`** (§3); `RecipeIngester`
  is a typed interface with a `HowToCookIngester` stub and a `RecipeSink` write
  endpoint (§4); the recipe agent is a thin stub (§5). Stubs are fully typed; absent
  behavior raises `NotImplementedError`.
- An AgentOS instance is constructed with the placeholder agent registered;
  AgentOS's built-in `/health` probe responds.
- v0.1 gates stay green on the new modules: `ruff` / `mypy` (strict) / `pytest`,
  coverage ≥ 80%. Tests assert only stub *shape and typing* — no DB, network, or
  LLM.

## 1. Recipe Module Structure

The recipe agent and its domain (model, ingestion) live in a self-contained
`recipe_agent/` package. The top level — alongside `app.py`, the composition
root — is reserved for **Runtime-level shared components** (Profile Management,
database, knowledge, model config); none are created in v0.2 (all out of scope),
but the layout leaves them an obvious home rather than mixing them into an agent
package.

```
src/spectres_runtime/
├── app.py                     # composition root: AgentOS + built-in /health (extends v0.1)
└── recipe_agent/
    ├── __init__.py
    ├── agent.py               # recipe agent construction (thin stub)
    ├── models/                # typed domain model
    │   ├── __init__.py        # re-exports Recipe, Ingredient, IngredientRole, RecipeProvenance
    │   ├── recipe.py          # Recipe
    │   ├── ingredient.py      # Ingredient + IngredientRole enum
    │   └── provenance.py      # RecipeProvenance
    └── ingestion/
        ├── __init__.py
        ├── ingester.py        # RecipeIngester interface
        ├── howtocook.py       # HowToCookIngester stub
        └── sink.py            # RecipeSink + WriteResult (shared write endpoint)
(future siblings of recipe_agent/: config.py (top-level); storage/ for DB wiring
 — db.py + knowledge.py; profile/ for domain logic. Grouping settled in v0.3 §4.)
tests/recipe_agent/
├── test_agent.py              # agent construction + AgentOS registration
├── test_models.py             # Recipe shape / typing
└── test_ingestion.py          # ingestion shape; stubs raise NotImplementedError
```

- **One package per agent.** Future agents (health, shopping) are siblings of
  `recipe_agent/`; shared components never live under an agent.
- **`recipe_agent/` holds the agent *and* its domain** (model, ingestion) — single
  consumer for now. The domain promotes to a top-level shared package when a
  second agent needs it.
- `models/` is a **package** (one module per concept), and `ingestion/` is a
  **package** too — additional ingesters are future siblings of `howtocook.py`, so
  the directory exists from day one.

**Deliverables:**

- [x] `recipe_agent/` subpackage + `tests/recipe_agent/` mirror; all modules
  importable.

## 2. AgentOS Wiring (Minimal)

Stand up an AgentOS instance and register the placeholder agent — no model
traffic, no database.

- Construct an `AgentOS` and expose its app, keeping the module-level `app` so
  `uvicorn spectres_runtime.app:app` works. AgentOS ships a built-in `/health`
  liveness probe, so no custom probe route is added.
- Static construction at module load — sufficient for one built-in stub agent;
  `AgentFactory` / directory-driven construction deferred.
- Register the recipe agent (§5) so it is discoverable on the AgentOS HTTP
  surface. It need not *do* anything yet.
- **No `PostgresDb`, `knowledge`, `dependencies`, or tools** wired here. A model
  is configured only as far as constructing an agent requires, and is **never
  invoked** (see Resolved Questions).

> **Probe change from v0.1.** v0.1 served a custom `/healthz` returning
> `{"status": "ok"}`. AgentOS ships its own `/health` probe (same `status: ok`,
> plus an `instantiated_at` timestamp), so v0.2 drops the redundant custom route
> and standardizes on `/health`. The v0.1 plan's `/healthz` references are left
> as a historical record.

**Deliverables:**

- [x] AgentOS serving, built-in `/health` responds; placeholder agent registered.

## 3. `Recipe` Domain Model (Typed Stub)

Define the model from
[design §3](../design/recipe-agent.md#3-domain-model-recipe) as **types only** — no
parsing, no validation beyond what the type system gives for free.

- Core fields: `id`, `name`, `aliases`, `description`, `images`, `category`,
  `steps`, `servings`, `difficulty`, `time`, and `provenance`.
- **Free-text fields hold Markdown by contract.** `description` and `steps` are
  Markdown strings (`steps` may carry `###` phase subsections, preserved as-is);
  non-Markdown sources convert during normalization. `steps` is a single string,
  not a list — it has no programmatic per-step consumer in scope, and raw
  Markdown is lossless.
- **Typed scalars, not free text.** `difficulty` is an `int` (1-5); `time` is a
  `float` in hours; both optional. `servings` is an optional `int`.
- `images` is a `list[str]` of runtime-resolvable references (local path or
  served URL), populated at ingestion — not the upstream address.
- `provenance` (`RecipeProvenance`) records origin: `source` (the source name)
  plus an optional `ref` (original id / URL within that source).
- **Structured `ingredients`** — a list of `{name, quantity, unit, role,
  optional}` with `role ∈ {main, supporting, seasoning}` (an enum) and
  `optional: bool`. Committing this shape now is what later makes portion scaling
  and filtering possible. `quantity` is a raw **string** so ranges (e.g. `10-15`)
  survive losslessly; quantity/unit parsing is not implemented, only the field
  shape.

**Deliverables:**

- [x] `Recipe` (+ `Ingredient`, `role` enum) defined and type-checked.

## 4. Ingestion Layer (Typed Stubs)

Define the ingestion boundary from
[design §2.1](../design/recipe-agent.md#2-data-source-strategy) — the per-origin
ingester interface, a non-functional first implementation, and the shared write
endpoint. There is **no** custom query/`RecipeSource` interface: read-back is Agno's
native `search_knowledge_base` (design §5), so v0.2 stubs only the *write* path.

- **`RecipeIngester`** (`ingester.py`) — the typed contract every origin honors:
  a `name` class var plus `ingest() -> Iterator[Recipe]`. Owns the origin's
  transport; later phases depend on this interface, not any implementation.
- **`HowToCookIngester`** (`howtocook.py`) — implements the interface
  (`name = "howtocook"`); `ingest()` raises `NotImplementedError`. No vendoring,
  parsing, or embedding.
- **`RecipeSink`** (`sink.py`) — the shared write endpoint, decoupled from
  ingesters: `write(recipes: Iterable[Recipe]) -> WriteResult` (counts
  written/skipped/failed). Persistence is identical across origins, so it lives
  here once; the body raises `NotImplementedError`.

**Deliverables:**

- [x] `RecipeIngester` interface + `RecipeSink`/`WriteResult` defined;
  `HowToCookIngester` stub raises `NotImplementedError`.

## 5. Minimal Recipe Agent (Skeleton)

A thin `agent.py` that *names* the composition from
[design §5](../design/recipe-agent.md#5-agent-composition--rag) without wiring its
parts.

- A single construction entry point (factory function or module-level `Agent`)
  that AgentOS (§2) registers, establishing *where* `model`, `knowledge`, `db`,
  `tools`, and `instructions` will attach.
- Those collaborators are unwired placeholders — no retrieval, no User Memory; the
  agent answers no user journeys yet.
- Bar: the module imports, constructs without error, and the app boots with it
  registered.

**Deliverables:**

- [x] `agent.py` construction entry point registered; app boots, `/health` green.

## Explicitly Out of Scope for v0.2

Everything operational is deferred — v0.2 is *only* the typed skeleton.

- **Postgres + pgvector wiring** — no `PostgresDb`, no `Knowledge(vector_db=...)`,
  no embeddings.
- **Containerization** — no `Dockerfile` / `docker-compose.yml`.
- **HowToCook ingestion** — no vendoring, Markdown→`Recipe` parsing, or loading;
  `HowToCookIngester` stays a stub.
- **Retrieval / RAG and the agent's user journeys** — the agent answers nothing.
- **Live model traffic** — the configured model is never invoked.
- **All personalization / User Profile work** — Profile Management, `User` /
  `Household`, merged constraints, `dependencies` injection, tools, User Memory.
- **Data sources beyond the HowToCook stub** (design §2 Phase B/C).
- **Auth / JWT / RBAC**, **Alembic migrations**, **media / object store**, **cloud
  deployment / CD**, **Edge / Client integration**.

## Resolved Questions

- **Does a registered AgentOS agent need a constructed `model`?** Yes. AgentOS
  injects a default provider model (pulling its SDK) when none is given, so a
  fully unwired placeholder is not possible. Resolved with a `_PlaceholderModel`
  that constructs without any SDK and raises `NotImplementedError` on every
  invocation — never called in v0.2. (§2 / §5.)
- **`Recipe` representation: Pydantic vs. dataclass?** Pydantic `BaseModel` —
  typed validation for free and it matches the framework the agents are built on.
