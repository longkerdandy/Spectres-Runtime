# v0.2 — Recipe Agent Skeleton

**Status:** planning draft. Architecture and rationale live in the design doc
[`docs/design/recipe-agent.md`](../design/recipe-agent.md); this is the v0.2 work
breakdown. Runtime-wide concepts (AgentOS, Postgres + pgvector) are defined in
[`Agents.md`](../../Agents.md).

**Goal:** Build the recipe agent's **code skeleton** — nothing more. v0.2 adds the
`recipe/` module layout and *typed stub definitions* (the `Recipe` model, the
`RecipeSource` interface, a `HowToCookSource` stub, and a thin recipe-agent
module), and registers a placeholder agent into an
[AgentOS](https://docs.agno.com/agent-os/introduction) instance so the app still
boots.

Like v0.1 ("engineering-complete but business-empty"), v0.2 is
**"structurally-complete but behavior-empty"**: the modules exist, are typed,
import cleanly, and the app boots — but nothing is parsed, embedded, retrieved, or
answered. Database wiring, ingestion, RAG, real agent behavior, and **all user
profile / personalization work** are deferred (see Out of Scope).

**Definition of Done:**

- `src/spectres_runtime/recipe/` exists with the §1 layout and imports cleanly.
- `Recipe` is a typed model with **structured `ingredients`** (§3); `RecipeSource`
  is a typed interface with a `HowToCookSource` stub (§4); the recipe agent is a
  thin stub (§5). Stubs are fully typed; absent behavior raises
  `NotImplementedError`.
- An AgentOS instance is constructed with the placeholder agent registered;
  `/healthz` still passes.
- v0.1 gates stay green on the new modules: `ruff` / `mypy` (strict) / `pytest`,
  coverage ≥ 80%. Tests assert only stub *shape and typing* — no DB, network, or
  LLM.

## 1. Recipe Module Structure

New code lives in a `recipe/` subpackage so the domain has a clear home and later
releases extend it in place.

```
src/spectres_runtime/
├── app.py                 # AgentOS instance + /healthz (extends v0.1)
└── recipe/
    ├── __init__.py
    ├── models.py          # Recipe (+ Ingredient, role enum) — typed model
    ├── sources/
    │   ├── __init__.py
    │   ├── base.py        # RecipeSource interface
    │   └── howtocook.py   # HowToCookSource stub
    └── agent.py           # recipe agent construction (thin stub)
tests/recipe/
├── test_models.py         # Recipe shape / typing
└── test_sources.py        # interface contract; stub raises NotImplementedError
```

- `sources/` is a **package**, not a module — Phase B/C sources are future
  siblings of `howtocook.py`, so the directory exists from day one.

**Deliverables:**

- [ ] `recipe/` subpackage + `tests/recipe/` mirror; all modules importable.

## 2. AgentOS Wiring (Minimal)

Stand up an AgentOS instance and register the placeholder agent — no model
traffic, no database.

- Construct an `AgentOS` and expose its app, keeping the module-level `app` so
  `uvicorn spectres_runtime.app:app` and existing probes work and `/healthz` is
  retained.
- Static construction at module load — sufficient for one built-in stub agent;
  `AgentFactory` / directory-driven construction deferred.
- Register the recipe agent (§5) so it is discoverable on the AgentOS HTTP
  surface. It need not *do* anything yet.
- **No `PostgresDb`, `knowledge`, `dependencies`, or tools** wired here. A model
  is configured only as far as constructing an agent requires, and is **never
  invoked** (see Open Questions).

**Deliverables:**

- [ ] AgentOS serving, `/healthz` retained; placeholder agent registered.

## 3. `Recipe` Domain Model (Typed Stub)

Define the model from
[design §3](../design/recipe-agent.md#3-domain-model-recipe) as **types only** — no
parsing, no validation beyond what the type system gives for free.

- Core fields: `id`, `name`, `aliases`, `category` / `cuisine`, `steps`,
  `servings`, optional `difficulty` / `time`, and `source` provenance.
- **Structured `ingredients`** — a list of `{name, quantity, unit, role}` with
  `role ∈ {main, aux, seasoning}` (an enum). Committing this shape now is what
  later makes portion scaling and filtering possible; quantity/unit parsing is not
  implemented, only the field shape.

**Deliverables:**

- [ ] `Recipe` (+ `Ingredient`, `role` enum) defined and type-checked.

## 4. `RecipeSource` Interface + `HowToCookSource` Stub

Define the pluggable boundary from
[design §2](../design/recipe-agent.md#2-data-source-strategy) — the interface, plus
a non-functional first implementation.

- **`RecipeSource`** (`base.py`) — the typed contract every source honors, roughly
  `search(constraints) -> list[Recipe]` and `get(id) -> Recipe`. Later phases
  depend on this interface, not any implementation.
- **`HowToCookSource`** (`howtocook.py`) — implements the interface; every method
  raises `NotImplementedError`. No vendoring, parsing, embedding, or loading.

**Deliverables:**

- [ ] `RecipeSource` interface defined; `HowToCookSource` stub raises
  `NotImplementedError`.

## 5. Minimal Recipe Agent (Skeleton)

A thin `agent.py` that *names* the composition from
[design §5](../design/recipe-agent.md#5-agent-composition--rag) without wiring its
parts.

- A single construction entry point (factory function or module-level `Agent`)
  that AgentOS (§2) registers, establishing *where* `model`, `knowledge`, `db`,
  `tools`, and `instructions` will attach.
- Those collaborators are unwired placeholders — no retrieval, no User Memory; the
  agent answers no user journeys yet.
- Bar: the module imports, constructs without error, and the app boots with it
  registered.

**Deliverables:**

- [ ] `agent.py` construction entry point registered; app boots, `/healthz` green.

## 6. Testing & Quality

Extend the v0.1 gates to the new modules; keep them green on stubs.

- `ruff` / `mypy` (strict) / `pytest` cover all new modules; coverage ≥ 80%.
- **Shape/typing tests only** — construct a `Recipe`, assert the `ingredients`
  shape and `role` enum; assert `HowToCookSource` methods raise
  `NotImplementedError`. No external services, network, or live LLM.
- `NotImplementedError` bodies are covered by the existing `exclude_lines` rule
  (v0.1 §3), so the coverage gate stays honest.

**Deliverables:**

- [ ] New modules pass all gates; coverage ≥ 80%; CI needs no LLM / DB / network.

## Explicitly Out of Scope for v0.2

Everything operational is deferred — v0.2 is *only* the typed skeleton.

- **Postgres + pgvector wiring** — no `PostgresDb`, no `Knowledge(vector_db=...)`,
  no embeddings.
- **Containerization** — no `Dockerfile` / `docker-compose.yml`.
- **HowToCook ingestion** — no vendoring, Markdown→`Recipe` parsing, or loading;
  `HowToCookSource` stays a stub.
- **Retrieval / RAG and the agent's user journeys** — the agent answers nothing.
- **Live model traffic** — the configured model is never invoked.
- **All personalization / User Profile work** — Profile Management, `User` /
  `Household`, merged constraints, `dependencies` injection, tools, User Memory.
- **Data sources beyond the HowToCook stub** (design §2 Phase B/C).
- **Auth / JWT / RBAC**, **Alembic migrations**, **media / object store**, **cloud
  deployment / CD**, **Edge / Client integration**.

## Open Questions

- Can a registered AgentOS agent be a fully unwired placeholder, or does it
  require a constructed `model`? (§2 / §5.)
- `Recipe` representation: Pydantic model vs. dataclass.
