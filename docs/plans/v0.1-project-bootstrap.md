# v0.1 — Development Environment Setup

**Goal:** Establish the engineering baseline for the Spectres-Runtime repository so that all subsequent feature work happens in a uniform, automated, reproducible environment. This release intentionally ships **no business functionality** (no AgentOS code, no agents, no database) — only an "empty but engineering-complete" project skeleton.

**Definition of Done:**

- Any contributor can `git clone` the repo and, by following `README.md`, bring up a runnable empty application and pass all checks (`lint` / `type` / `test`) within 10 minutes.
- On every branch push and pull request, GitHub Actions runs the same checks, and code-scanning results appear in the GitHub Security tab.
- Every commit conforms to Conventional Commits. Branching is trunk-based for v0.1; Git Flow is deferred until the project actually needs it (see §6 for the upgrade triggers).

## 1. Python Environment & Dependency Management

**Status: ✅ Completed.**

- **Python version:** pinned to `3.12` (uv-managed `cpython-3.12.13`). System Python is left untouched.
  - `.python-version` contains `3.12`.
  - `pyproject.toml` declares `requires-python = ">=3.12,<3.13"` (upper bound locked to keep wheel availability predictable until 3.13 ecosystem matures).
- **Package manager:** [`uv`](https://docs.astral.sh/uv/) (v0.11.17+) — fast, native `pyproject.toml` + lockfile support, compatible with the AgentOS ecosystem.
  - Project bootstrapped with `uv init --lib --build-backend hatch` (official template, `src/` layout, `py.typed` marker included).
  - Artifacts committed: `pyproject.toml`, `uv.lock`, `.python-version`.
- **Build backend:** `hatchling` (chosen by `uv init`'s `hatch` backend option; widely used, low-friction).
- **License declared via PEP 639 SPDX:** `license = "Apache-2.0"` (the `LICENSE` file itself ships in §10).
- **Runtime dependencies (`[project.dependencies]`) — placeholders, no wiring yet:**
  - `agno>=2.6.9` — will host AgentOS.
  - `fastapi>=0.136.3` — web entry point.
  - `uvicorn[standard]>=0.48.0` — ASGI server.
- **Dependency groups (PEP 735 `[dependency-groups]`):**
  - `lint` — `ruff` only. Kept minimal so the CI lint job stays sub-second.
  - `test` — `pytest`, `pytest-asyncio`, `pytest-cov`.
  - `dev` — `mypy`, `pre-commit`, plus `include-group = "lint"` + `include-group = "test"`. This is uv's default group, so `uv sync` (no flags) installs the full developer environment in one command.
- **Why three groups, not one:** lets CI install only what each job needs (`uv sync --only-group lint` for the lint job, `--only-group test` for tests), keeps production images slim (`uv sync --no-dev`), and gives each group a clear semantic role.
- **Why `ruff` alone in `lint`:** ruff already subsumes flake8 + isort + black + pyupgrade + bugbear + simplify + a baseline of bandit-style security rules (`S`). Deep security scanning is handled by CodeQL (§8); CVE scanning by Dependabot (§8). Additional lint tools (codespell / vulture / interrogate) can be added incrementally if a concrete need appears.
- **Standard developer commands (to be documented in `README.md` under §10):**

  ```bash
  uv python install 3.12        # one-time, if uv hasn't already
  uv sync                       # installs runtime + dev (= lint + test + mypy + pre-commit)
  uv run python -c "import spectres_runtime; print(spectres_runtime.hello())"
  ```

- **CI-oriented commands (to be wired in §7):**

  ```bash
  uv sync --frozen --only-group lint     # lint job
  uv sync --frozen --only-group test     # test job
  uv sync --frozen --no-dev              # production / container image
  ```

  - `.python-version`, `pyproject.toml`, `uv.lock`.
  - `src/spectres_runtime/__init__.py` + `src/spectres_runtime/py.typed` (placeholder package; real modules in §2).
  - `.venv/` reproducible via `uv sync --frozen` (excluded by `.gitignore`).

## 2. Empty Project Skeleton

**Status: ✅ Completed.**

Minimal runnable structure, to be extended by later releases:

```
src/
└── spectres_runtime/
    ├── __init__.py     # package metadata (__version__)
    ├── py.typed        # PEP 561 marker (shipped by uv init --lib)
    └── app.py          # FastAPI() instance + /healthz endpoint + main()
tests/
└── test_healthz.py     # smoke test via fastapi.testclient.TestClient
```

- **Layout:** `src/` layout (from `uv init --lib`), prevents implicit imports and forces installation through the build backend.
- **FastAPI app:** module-level `app: Final[FastAPI]` so `uvicorn spectres_runtime.app:app` works directly. Title/version/description set from project metadata.
- **`/healthz` endpoint:** async, no I/O, returns `{"status": "ok"}`. Serves as both a skeleton self-check and a probe for CI / container health checks. Tagged `meta` so it's grouped separately from future business endpoints in the OpenAPI schema.
- **`main()` callable:** kept in `app.py` for a future `[project.scripts]` entry point. **Not** registered in `pyproject.toml` for v0.1 — `uvicorn spectres_runtime.app:app` is the standard launch command; scripts entry can be added later without breaking anything.
- **Smoke test:** uses `fastapi.testclient.TestClient` (synchronous wrapper, no event loop boilerplate). One test asserts status `200` and exact JSON payload. Verified passing with `uv run pytest`.
- **Known warning (non-blocking):** Starlette's `TestClient` emits `StarletteDeprecationWarning: Using httpx with starlette.testclient is deprecated; install httpx2 instead`. Test still passes; `httpx2` is too new to adopt. Suppressed in §3 via pytest `filterwarnings`.
- **Launch verified:** `uv run uvicorn spectres_runtime.app:app` boots successfully; `GET /healthz` returns `{"status": "ok"}`; `/openapi.json` auto-generated with the single endpoint listed.

## 3. Code Quality Tool Configuration

**Status: ✅ Completed.**

All configuration lives in `pyproject.toml` under `[tool.*]`; no scattered `setup.cfg` / `.flake8` / `mypy.ini`. All three tools verified passing after install (`uv run ruff check . && uv run ruff format --check . && uv run mypy && uv run pytest`).

- **Ruff** (`[tool.ruff]` + `[tool.ruff.lint]` + `[tool.ruff.format]`)
  - `line-length = 100`, `target-version = "py312"`.
  - Rule sets enabled: `E, W, F, I, B, UP, SIM, RUF` — the consensus modern-Python baseline used by FastAPI / httpx / pydantic. Kept intentionally small; expand only when a concrete need appears.
  - `per-file-ignores`: `tests/**/*.py` is allowed `S101` (`assert`) since pytest's model is built on it.
  - Formatter aligned with Black defaults: double quotes, spaces, LF — explicit choices keep cross-platform diffs stable.
- **Mypy** (`[tool.mypy]`)
  - `python_version = "3.12"`, `strict = true` (bundles ~10 individual checks).
  - `files = ["src", "tests"]` — type-checks both production code and tests.
  - `show_error_codes = true`, `pretty = true` — error codes enable surgical `# type: ignore[code]` instead of blanket ignores.
  - Extra warn flags on: `warn_unused_ignores`, `warn_redundant_casts`, `warn_return_any`.
  - Override for `tests.*`: `disallow_untyped_decorators = false` so untyped third-party pytest decorators don't require stubs.
- **Pytest** (`[tool.pytest.ini_options]`)
  - `minversion = "9.0"`, `testpaths = ["tests"]`.
  - `addopts`: `-ra`, `--strict-markers`, `--strict-config`, `--cov=spectres_runtime`, `--cov-report=term-missing`, `--cov-fail-under=80`.
  - `filterwarnings`: suppresses Starlette's environmental `StarletteDeprecationWarning` (httpx → httpx2 nudge).
  - `asyncio_mode = "auto"` — `async def test_*` works without per-test markers.
- **Coverage.py** (`[tool.coverage.run]` + `[tool.coverage.report]`)
  - `source = ["src/spectres_runtime"]`, `branch = true` — both edges of conditionals must be exercised.
  - `exclude_lines`: `pragma: no cover`, `if __name__ == "__main__":`, `raise NotImplementedError`.
  - `show_missing = true`, `skip_covered = false` so all files appear in the report.
- **Coverage gate met:** baseline 100% (`__init__.py` + `app.py`); `main()` excluded via `# pragma: no cover` as a thin uvicorn wrapper exercised manually, not in unit tests.
- **Comment hygiene:** `pyproject.toml` comments explain *why* and *what* (decisions, semantics); status-shaped comments (`yet` / `for now` / `will be` / `once`) are avoided to prevent rot as the project evolves.

**Status: ✅ Completed.**

- **Hooks installed** (in stage order):

Carries `additional_dependencies` for `fastapi`, `uvicorn[standard]`, and `agno` because each pre-commit hook runs in its own isolated venv and can't see the project `.venv`. CI runs the full mypy invocation (src + tests).
  - **`compilerla/conventional-pre-commit` v4.3.0** — runs on the `commit-msg` stage; rejects commit messages that don't follow Conventional Commits.
- **Installation (one-time per clone):**

  ```bash
  uv run pre-commit install                          # .git/hooks/pre-commit
  uv run pre-commit install --hook-type commit-msg   # .git/hooks/commit-msg
  ```

- **Manual full-repo run:** `uv run pre-commit run --all-files` — all hooks pass on the current skeleton.
- **Commit-msg gate verified:** ad-hoc tests with a bad message ("added stuff") fail; a Conventional Commits message ("chore: add pre-commit config") passes.
- **CI vs. pre-commit boundary:** the two run the same checks but along different paths — pre-commit uses isolated per-hook venvs for speed and runs on staged files, CI uses the project `.venv` and runs on the full tree. This is intentional; see §7 for the CI side.

## 5. Commit Message Conventions

**Status: ✅ Completed.**

- **Convention:** [Conventional Commits 1.0](https://www.conventionalcommits.org/).
  - Allowed `type` values: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`.
  - `scope` is optional; lowercase module / area names recommended (e.g. `agent`, `memory`, `ci`, `deps`).
  - `subject` style: imperative mood, lowercase, no trailing period, ≤ ~50 chars.
- **Single source of truth: `CONTRIBUTING.md` at the repo root.** Houses both the machine-enforced rules and the project conventions, split into two clearly-labelled subsections so contributors know which lines a tool will actually reject:
  - **Machine-enforced** (what the `commit-msg` hook from §4 validates): type whitelist, `type(scope): subject` shape, `!` breaking-change marker, non-empty subject, blank line before body.
  - **Project conventions** (NOT machine-enforced, kept as a written agreement for humans and AI coding assistants): lowercase subject, ≤ 50 chars, imperative mood, no trailing period, scope naming, ~72-char body wrap, footer keywords (`BREAKING CHANGE`, `Refs`, `Closes`).
- **`Agents.md` carries a pointer, not the rules.** A short `## Repository Conventions` section at the bottom of `Agents.md` instructs AI coding assistants (which auto-load `Agents.md` into their system prompt) to read `CONTRIBUTING.md` before making changes. This keeps `Agents.md` focused on *what the system builds* and avoids the "Coding Agents vs. Agno `Agent`" naming collision that arose from earlier drafts.
- **No `.gitmessage` template.** An editor-pre-fill template was prototyped and removed: AI agents commit via `git commit -m`, which bypasses `commit.template` entirely, and the project is single-developer with no editor-driven commit workflow. Keeping a template would create a second source of truth that drifts from `CONTRIBUTING.md` without adding enforcement. The previous `git config --local commit.template` is unset.
- **Enforcement layers:**
  - **Local:** `commit-msg` pre-commit hook (`compilerla/conventional-pre-commit`, configured in §4) rejects structurally non-conforming messages.
  - **CI:** PR title validation via `amannn/action-semantic-pull-request` (planned in §7).
  - **None of the above enforces the "Project conventions" tier** — that gap is acknowledged in `Agents.md` and accepted for v0.1.
- **README quick-start** covers §1–§5 commands in one block: `uv python install 3.12`, `uv sync`, `pre-commit install` (×2 stages), then smoke tests. Full README polish is deferred to §10.

## 6. Branching Strategy

**Status: ✅ Completed (tag deferred until §1–§10 all green).**

**Decision: trunk-based for v0.1.** Git Flow was the original plan but was rejected after re-evaluation — its assumptions (multiple contributors, parallel maintained versions, scheduled release windows, mandatory peer review) do not hold for a single-developer pre-alpha project where the second "contributor" is an AI coding assistant. Forcing PR + review on a one-person repo adds 5–10× friction with no quality gain, and the original §6 requirement "release v0.1.0 by going through the Git Flow release flow" was a circular dependency (using a process that this release was meant to establish).

**Model — trunk-based, documented in `CONTRIBUTING.md`:**

- A single long-lived branch: **`main`**. All work lands on `main` directly.
- Short-lived topic branches are optional, used only when an experiment is large enough to want isolation; they merge back to `main` via fast-forward or squash.
- Releases are **annotated tags** on `main`: `git tag -a v0.1.0 -m "v0.1.0" && git push --tags`. No `release/*` branch.
- Hotfixes are normal commits on `main`, tagged with a new patch version. No `hotfix/*` branch.

**Branch protection (GitHub Settings) — intentionally minimal:**

- Forbid force-push to `main`.
- Forbid deletion of `main`.
- **Allow direct push to `main`.** No required reviewers, no required PR.
- CI status (§7) runs on every push and tag but is not a merge blocker (there is nothing to block).

**Tags:** semantic versioning `vMAJOR.MINOR.PATCH`. This release ships as `v0.1.0`.

**Upgrade triggers — when to revisit and adopt Git Flow (or GitHub Flow):**

- A second permanent human contributor joins the repo.
- A production deployment exists and needs to be maintained separately from `main`'s in-progress work (i.e. a real reason for a `develop` branch).
- Two or more versions must be supported in parallel (i.e. a real reason for `hotfix/*`).

Until at least one of the above is true, the cost of Git Flow exceeds its benefit. The decision is captured in `CONTRIBUTING.md` so future contributors understand the rationale rather than assume it was an oversight.

## 7. GitHub Actions

**Status: ✅ Completed.**

Directory: `.github/workflows/`.

- **`ci.yml`** — triggers: `push` to `main` and any `pull_request`. Tag pushes intentionally do **not** trigger CI: a tag points at a commit that `push: main` has already verified, and tag-driven work belongs to a future `release.yml` (deferred to ≥ v0.2). This keeps `ci.yml` = continuous verification and `release.yml` = release actions cleanly separated.
  - Single Python version: `3.12`. No matrix in v0.1 because `requires-python = ">=3.12,<3.13"` forbids 3.13; a "reserved slot" would only produce noise.
  - Steps: `actions/checkout` → install `uv` (`astral-sh/setup-uv`, cache enabled, keyed on `uv.lock`) → `uv sync --frozen` → `uv run ruff check .` → `uv run ruff format --check .` → `uv run mypy` → `uv run pytest`.
  - `concurrency` group keyed on `github.ref` with `cancel-in-progress: true` so repeated pushes do not stack runs.
  - `permissions: contents: read` — least privilege.
- **`pr-title.yml`** — uses `amannn/action-semantic-pull-request` to enforce Conventional Commits in PR titles. Only runs when a PR is opened; the `commit-msg` hook (§4) is the primary enforcement for direct-to-`main` commits.
- **`codeql.yml`** — GitHub-native code scanning; see §8.
- **(Optional, not required for v0.1) `release.yml`** — auto-generates GitHub Release notes on tags pushed to `main`; defer polish to v0.2.

All workflows pin third-party actions to specific versions (never `@main`).

## 8. GitHub Code Scanning & Dependency Security

**Status: ✅ Completed.**

GitHub-native only — no third-party SaaS, no extra dashboards. The repository is public, so every item below is zero-cost.

- **CodeQL** (`.github/workflows/codeql.yml`): Advanced setup (workflow file in the repo, not GitHub-managed default setup) for consistency with `ci.yml` / `pr-title.yml`. Language `python`, build-mode `none` (interpreted, no compile step). Triggers: `push` to `main`, PRs, and a weekly cron. Results surface under **Security → Code scanning alerts**. Not a merge blocker — fits trunk-based which does not use required status checks.
- **Dependabot Version Updates** (`.github/dependabot.yml`): ecosystems `pip` (uv lockfile is natively supported) and `github-actions`, weekly cadence, default target branch (`main`). Commit prefix: `chore(deps)` (and `chore(deps-dev)` for dev-group deps) to match the Conventional Commits examples in `CONTRIBUTING.md`. No grouping in v0.1 — revisit if PR volume becomes noisy. Labels: `dependencies`.
- **Dependabot Security Updates**: enabled by default for public repos; no config file needed. Auto-opens PRs for known-CVE upgrades regardless of the version-update schedule.
- **Dependency Review** (`.github/workflows/dependency-review.yml`): runs on PRs only; uses `actions/dependency-review-action` to block PRs that introduce dependencies with known vulnerabilities. Complements Dependabot — Dependabot upgrades *existing* deps, Dependency Review gates *new* deps at PR time.
- **Secret Scanning + Push Protection**: enabled via `gh api` (the GitHub Settings UI exposes the same toggles). Public-repo free tier. No workflow file — repository-level setting.
- **Private Vulnerability Reporting**: enabled via `gh api`. Gives outside researchers a private channel to report security issues instead of public-issue disclosure.
