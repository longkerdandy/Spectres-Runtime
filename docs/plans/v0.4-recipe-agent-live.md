# v0.4 — Recipe Agent Goes Live (Grounded Conversational RAG)

**Status:** planning draft. Architecture and rationale live in the design doc
[`docs/design/recipe-agent.md`](../design/recipe-agent.md); this is the v0.4 work
breakdown. The prior data slice is
[`v0.3-recipe-ingestion.md`](./v0.3-recipe-ingestion.md). Runtime-wide concepts
(AgentOS, Models, Postgres + pgvector) are defined in [`Agents.md`](../../Agents.md).

**Goal:** Give the v0.3 data layer its **first conversational behavior**. v0.4
replaces the placeholder model with a **live domestic chat model** (Moonshot / Kimi),
attaches the v0.3 **knowledge base** so the agent retrieves recipes through Agno's
native agentic RAG (`search_knowledge_base`), wires a **`PostgresDb`** for
**multi-turn sessions** (conversation history + context), and gives the agent
**grounding instructions** — so it answers Chinese recipe questions, in context,
grounded in *retrieved* recipes. This is the first time the runtime issues **real
LLM traffic**.

Where v0.3 was *"data-complete but conversation-empty"*, v0.4 is
**"conversation-complete but journey-empty"**: the agent talks, remembers the
current conversation, and stays grounded — but it only answers **ad-hoc recipe
questions**. The structured **meal-planning journeys** (design §6) and **preference
learning** (User Memory) land in v0.5; **all user-profile / household
personalization** (design §4) is a **separate module**, deferred beyond v0.5 (see
Out of Scope).

**Definition of Done:**

- A live chat model is configured and reachable (§1–§2): a `CHAT_API_KEY` is wired
  through `Settings`, and `build_recipe_agent()` carries a real
  `agno.models.moonshot.MoonShot` instead of the placeholder.
- The agent retrieves over the v0.3 corpus (§3–§4): with `knowledge` attached, Agno
  enables `search_knowledge_base`; a Chinese recipe question returns a **non-empty,
  grounded** answer that reflects a retrieved recipe — not the model's parametric
  memory.
- **Multi-turn works** (§3, §6): with `db` attached and `add_history_to_context=True`,
  a follow-up that refers to a previous turn (e.g. 「那它要炒几分钟？」) resolves using
  conversation history; the session persists across runs in Postgres.
- The full path is exercised through **AgentOS's own run endpoint** (§5–§6):
  `POST /agents/recipe/runs` returns a grounded answer; the session surfaces on the
  AgentOS Sessions endpoints.
- Telemetry stays **off** end to end (§5): no anonymous run data leaves the process.
- Gates stay green on the **two-tier split** from v0.3 §10 (§7): unit tests assert
  agent **composition shape** with **no live model call** (coverage ≥ 80%); the live
  answer + multi-turn checks are `integration`, opt-in, and excluded from default CI.

## 1. Chat Model Provider (hosted, domestic LLM — Moonshot / Kimi)

The chat model is a **hosted, OpenAI-compatible domestic provider**, per
[`Agents.md` → Models](../../Agents.md) (the Runtime leans DeepSeek / Zhipu / Qwen /
Moonshot to keep inference in-region and cheap). v0.4 uses **Moonshot / Kimi** —
the provider v0.3 already named as the v0.4 chat model.

- **Chat model ≠ embedder.** The Kimi chat model has **no embeddings endpoint**; it
  only generates text. RAG still embeds queries with the v0.3 embedder
  (`Qwen/Qwen3-Embedding-0.6B` on SiliconFlow). The two are **separate providers,
  separate keys, separate base URLs** — do not conflate `CHAT_*` and `EMBEDDER_*`.
- **Agno has a first-class class.** `from agno.models.moonshot import MoonShot`
  (`MoonShot(id=..., api_key=..., base_url=...)`, a thin `OpenAILike` subclass). The
  wiring stays **provider-agnostic**: the model id, base URL, and key all come from
  config, so swapping to DeepSeek / Zhipu / Qwen (or another Kimi tier) is a
  **config change**, not a code change.
- **Provider channel: reuse the Kimi Code subscription.** v0.4 runs against an existing
  **Kimi Code** subscription, which exposes an **OpenAI-compatible** coding endpoint that
  `MoonShot` consumes unchanged. This is a **config decision**, not a code one — see the
  base-URL and model-id bullets below. (Kimi Code is positioned for coding agents and has
  subscription rate limits; the **Moonshot open platform** — `https://api.moonshot.cn/v1`,
  pay-as-you-go — is the official channel for product integration and remains a one-line
  `.env` swap when needed.)
- **Base URL.** No code default — the base URL is a required `Settings` field read from
  `.env`. The **recommended value in `.env.example`** is the Kimi Code coding endpoint
  `https://api.kimi.com/coding/v1` (OpenAI-compatible). This must be set explicitly — note
  Agno's `MoonShot` class itself defaults to the `.ai` open-platform host, which the config
  value overrides. (Open-platform alternative if not using Kimi Code: `https://api.moonshot.cn/v1`,
  the `.cn` host — same SiliconFlow-style `.cn` vs `.ai`/`.com` key-namespace gotcha.)
- **Model id is config, not a data contract.** Unlike the embedder (whose id pins the
  stored vector space — v0.3 §1), the chat model id can change freely between runs; it
  governs only generation quality/cost, not stored data. Like every other field, it has
  **no code default** — it is a required `Settings` field from `.env`. The **recommended
  value in `.env.example`** is `kimi-for-coding` — the **unified model id Kimi Code
  exposes** (it does not surface the individual tiers). On the open platform the
  equivalent would be a specific tier such as `kimi-k2-thinking`; the exact tier /
  context window is confirmed against real queries at §6 (see Open Questions).

**Deliverables:**

- [ ] `MoonShot` chat model constructed from config, provider-agnostic; documented as
  distinct from the embedder.

## 2. Configuration & Dependencies

Extend the v0.3 `Settings` with the chat-model fields, mirroring the embedder block.
The **connection fields** (`chat_model`, `chat_base_url`, `chat_api_key`) follow the v0.3
rule — **no defaults, every value from the process env or the git-ignored `.env`**.
`chat_temperature` is the **one deliberate exception**: an optional tuning knob that may
fall back to the provider default.

- **New `Settings` fields:** `chat_model: str`, `chat_base_url: str`,
  `chat_api_key: SecretStr` (all required, no default) — plus `chat_temperature: float |
  None = None`, the **only optional field**, so an unset temperature means "use the
  provider default" rather than forcing a value. A `build_chat_model() -> MoonShot`
  method mirrors `build_embedder()`: it unwraps the secret, constructs the `MoonShot`
  client, and passes `temperature` **only when it is not `None`** (so `None` never
  overrides the provider default).
- **Temperature caveat (verify at §6):** low temperature favors grounding/reproducibility
  (relevant to the grounding-strictness Open Question), so a value like `0.3` is a
  reasonable `.env.example` recommendation. But confirm `kimi-for-coding` actually
  honors `temperature` — some coding / "thinking" tiers ignore or reject it; if so,
  leave it unset.
- **`.env.example`** documents the new keys (Kimi Code recommended values —
  `CHAT_BASE_URL=https://api.kimi.com/coding/v1`, `CHAT_MODEL=kimi-for-coding` — plus the
  open-platform alternative as a comment) alongside the existing `DATABASE_*` /
  `EMBEDDER_*` keys. **(Deferred to the full §2 pass — not done in the config-only slice.)**
- **Dependency:** no new third-party package — `MoonShot` ships with `agno`. (Confirm
  `uv.lock` already resolves it; otherwise `uv add` is not needed.)
- **`AGNO_TELEMETRY=false`** is added to `.env.example` as the global opt-out (see §5).

**Deliverables:**

- [ ] `Settings` carries `chat_*` fields + `build_chat_model()`; `.env.example`
  updated; config unit tests cover load + mapping (no network).

## 3. Agent Composition: model + knowledge + db + multi-turn

Turn `build_recipe_agent()` from a placeholder shell into the real composition from
[design §5](../design/recipe-agent.md#5-agent-composition--rag) — *without* the
personalization legs (deferred). The factory now takes `Settings`.

- **`model`** — `settings.build_chat_model()` (the real `MoonShot`). The
  `_PlaceholderModel` is **removed**.
- **`knowledge`** — `build_knowledge(settings)` (v0.3). Attaching `knowledge` makes
  Agno enable the built-in **`search_knowledge_base`** tool by default — the agent
  queries the corpus **on demand** (agentic RAG), the dataset is **not** auto-injected
  (design §5).
- **`db`** — `build_db(settings)` (the shared `PostgresDb`, v0.3). Drives **session
  persistence** and conversation history.
- **Multi-turn / context (confirmed scope):** `add_history_to_context=True` +
  `num_history_runs=N` (start N≈3–5) so each run sees the recent turns; `session_id`
  is supplied per request by AgentOS. History is **durable** in Postgres, so a session
  survives restarts and spans requests.
- **`telemetry=False`** on the `Agent` itself (not only on AgentOS — see §5).
- **Still unwired (deferred):** `dependencies` (household injection — §4), profile
  `tools`, and `learning`/User Memory. The factory establishes *where* they attach
  without wiring them.

**Deliverables:**

- [ ] `build_recipe_agent(settings)` wires real `model` + `knowledge` + `db` +
  multi-turn history; placeholder removed; `telemetry=False`.

## 4. Grounding Instructions & RAG Behavior

The agent is **thin** — its behavior is the model plus `instructions` plus
search-as-tool. v0.4 writes the minimal grounding instruction set from
[design §1](../design/recipe-agent.md#1-facts-vs-capability) (facts vs. capability):

- **Ground in retrieved recipes, never parametric memory.** Answer from what
  `search_knowledge_base` returns; if nothing relevant is retrieved, **say so** rather
  than inventing a recipe. The model *processes* retrieved recipes (adapts, explains),
  it does not *recall* them.
- **Answer in Chinese**, recipe-domain persona; cite the dish by name when grounding
  an answer so the grounding is visible/auditable.
- **Rely on Agno's native agentic search** (search-as-tool, on by default with
  `knowledge` attached) — **no custom retrieval store**, no `RecipeSource`, no
  whole-dataset injection (design §2.1 / §5).
- **Scope guard:** ad-hoc single-dish Q&A only. The instruction set does **not** yet
  describe meal-plan generation or plan editing (v0.5) and references **no** household
  / profile constraints (separate module).

**Deliverables:**

- [ ] Grounding instructions added; answers ground in retrieved recipes and degrade
  to "not found" instead of hallucinating.

## 5. AgentOS Wiring (config-driven construction; telemetry off)

AgentOS auto-generates its whole HTTP surface from what the agent declares — **we
write no endpoints**. "Integration depth" is therefore a function of which agent
fields §3 attaches. The routes already existed in v0.2/v0.3 (e.g. `POST
/agents/recipe/runs`), but hit the placeholder and raised `NotImplementedError`;
attaching a real `model` + `knowledge` + `db` makes them **functional**.

- **`build_app()` now needs `Settings`.** It constructs the agent from config and
  registers it. Consequence: the app's import-time construction now **requires
  configuration** (`.env` / env) and a reachable DB — note the implication for tests
  (§7) and document it. Construction stays **network-free** (no model/DB call at
  import; only on run).
- **Endpoint groups that go live in v0.4:** **Runs** (real LLM + agentic RAG,
  streaming + sync), **Sessions** (durable multi-turn, scoped by `user_id` /
  `session_id`), **Knowledge** (read/search over the 357-recipe corpus), **Traces /
  Metrics** (per-run token usage). Still **off / deferred:** **Memory**
  (no User Memory yet → v0.5), **Auth/JWT** (`authorization=True` left off —
  single-household, deferred module), **Evals**, **HITL / scheduling / background
  jobs**, **Teams / Workflows**, and the **os.agno.com control-plane UI** (opt-in,
  browser→runtime; v0.4 verifies via local HTTP / `/docs` only, never connects it).
- **Privacy posture (kept explicit):** the runtime is **self-hosted**; all session /
  knowledge / trace data lives only in the project's Postgres ("zero transmission" —
  nothing sent to Agno). The **one** default phone-home is Agno's anonymous telemetry
  to `api.agno.com` (model-provider / feature-flag metadata only — never prompts,
  responses, user data, or keys). It is disabled **two ways** for defence in depth:
  `telemetry=False` on **both** `AgentOS` (already set) **and** the `Agent` (§3), plus
  `AGNO_TELEMETRY=false` in the environment (§2).
- **Static construction stays.** One built-in agent at module load; `AgentFactory` /
  directory-driven construction remains deferred. Bump AgentOS `version` to `0.4.0`.

**Deliverables:**

- [ ] `build_app()` constructs the live agent from `Settings`; Runs/Sessions/Knowledge
  go live; telemetry off on AgentOS + Agent + env; `version="0.4.0"`.

## 6. Conversational Retrieval Verification

Prove the loop end to end **through AgentOS's run endpoint**, not just `agent.run()` —
so "AgentOS integration is functional" is actually demonstrated. These checks invoke
the live LLM and embedder, so they are `integration`-tier (opt-in, token-spending).

- **Grounded single-turn (via HTTP):** `POST /agents/recipe/runs` with
  「番茄炒蛋怎么做」 returns a **non-empty** answer that reflects the retrieved dish
  (e.g. mentions egg/tomato steps from `howtocook/vegetable_dish/西红柿炒鸡蛋`). The
  bar is **grounded + relevant**, not an exact string (LLM output is non-deterministic).
- **Multi-turn / context:** within one `session_id`, a first turn asks about a dish,
  a second turn refers back (「那它要炒几分钟？」 / 「换成不放葱呢？」) and the answer
  uses the prior turn — proving `add_history_to_context` works and the session
  persists. Assert the session is retrievable via the Sessions endpoint.
- **Skip-if-unconfigured:** like v0.3's integration checks, skip (not fail) when
  `CHAT_API_KEY` / DB are absent, so the tier is a no-op without credentials.

**Deliverables:**

- [ ] Integration test drives `POST /agents/recipe/runs` for a grounded Chinese
  answer **and** a 2-turn context-carrying exchange over one session.

## 7. Testing & Quality

Keep the **two-tier split** from v0.3 §10; v0.4 adds the first **chat-token** cost to
the integration tier and a new wrinkle: the app now needs config to construct.

- **Unit tier (default CI, pure).** No DB, no network, **no live model call**. Covers:
  `Settings` chat-field load/mapping; `build_chat_model()` returns a `MoonShot` mapped
  from config (constructed, never invoked), including the **optional-temperature paths**
  — asserts `temperature` is set when `chat_temperature` is provided and left at the
  provider default when it is `None`; `build_recipe_agent(settings)` **shape** —
  real model class attached, `knowledge` + `db` present, `instructions` non-empty,
  `add_history_to_context` set, `telemetry is False`. App construction is unit-tested
  with an **injected `Settings` double** (so it stays network-free and needs no `.env`).
- **Integration tier (`@pytest.mark.integration`, opt-in).** The §6 live HTTP run +
  multi-turn checks against the real Kimi API + pgvector. Excluded from the default
  `pytest` run and the coverage gate; skipped without credentials. **Token cost
  (new):** chat completions in addition to query embeddings.
- **Coverage honesty.** Integration-only glue (config-driven app construction, the
  `recipe-ingest` entry from v0.3) is scoped out of the unit denominator (per-file
  omit / `pragma`), per v0.3 §10 — the gate measures unit-testable code only.
- **CI stays token-free.** `ci.yml` keeps running bare `pytest` (`-m "not
  integration"`); no `CHAT_API_KEY`, embedder key, or DB in CI. The live tier is local
  / optional secret-gated, never on the default push path.

**Deliverables:**

- [ ] Unit tier green and pure (no live model), coverage ≥ 80%; integration tier
  covers the live grounded + multi-turn run, excluded from default CI, runnable
  locally.

## Explicitly Out of Scope for v0.4

v0.4 is the **"agent goes live as grounded conversational Q&A"** slice only.

- **Meal-planning user journeys** (design §6) — multi-day household **meal-plan
  generation** and **conversational plan adjustment** (a stateful plan object). → v0.5.
- **User Memory / preference learning** (Agno Learning Machine — "likes spicy",
  "found the braised pork too sweet"). Note this is **distinct** from Profile
  Management. → v0.5.
- **All user-profile / Profile Management work** (design §4) — `User` / `Household`,
  membership, **merged household constraints**, `dependencies` injection, member-detail
  / record-measurement **tools**, dietary restrictions / conditions / metrics. This is
  a **separate, independent module**, deferred beyond v0.5 (per project requirement).
- **Nutrition reasoning** / health-target matching (needs a nutrition-facts source).
- **Auth / JWT / RBAC** (`authorization=True`), **os.agno.com control-plane** linkage,
  **Teams / Workflows**, **`AgentFactory`** / directory-driven construction.
- **Retrieval upgrades** still deferred from v0.3: ANN index, Chinese FTS /
  `SearchType.hybrid`, reranking — add only if real v0.4 usage shows weak precision.
- **Media / object store** (image serving), **containerization / CD**, **Alembic
  migrations**, **Edge / Client**.

## Resolved Questions

- **Chat provider: Moonshot / Kimi, via the Kimi Code subscription.** Confirmed
  (requirement). OpenAI-compatible via Agno's `MoonShot` class; provider-agnostic wiring
  keeps a later swap (open platform / DeepSeek / Zhipu / Qwen) a config change. Channel:
  Kimi Code coding endpoint `https://api.kimi.com/coding/v1` with the unified
  `kimi-for-coding` model id.
- **Multi-turn included.** Confirmed (requirement): the agent carries **conversation
  context + durable history** via `db` + `add_history_to_context=True` +
  `num_history_runs`, with `session_id` managed per request by AgentOS. (Stateful
  *meal-plan* editing — a richer object than chat history — is still v0.5.)
- **AgentOS integration depth.** Static single built-in agent; we write no endpoints.
  Attaching `model`/`knowledge`/`db` makes **Runs / Sessions / Knowledge / Traces**
  functional; Memory / Auth / Evals / HITL / Teams / control-plane UI stay off. Verified
  via local HTTP (`TestClient` / `/docs`), never by connecting os.agno.com.
- **Self-hosted, no cloud upload.** Runtime data stays in the project's Postgres;
  AgentOS sends nothing to Agno. The only default phone-home is anonymous telemetry to
  `api.agno.com` (provider/feature metadata, no content) — disabled on AgentOS + Agent
  + via `AGNO_TELEMETRY=false`.
- **Chat model id is not a data contract.** It can change run-to-run (unlike the
  embedder id, which pins the stored vector space); only generation quality/cost
  depend on it.

## Open Questions

- **Exact Kimi tier + context window.** v0.4 reuses the **Kimi Code** subscription, whose
  unified `kimi-for-coding` id is the working value; whether its quality / context window
  suffices for grounded recipe Q&A (and whether switching to an open-platform tier such as
  `kimi-k2-thinking` is worthwhile) is settled against real queries at §6. Provider-agnostic
  wiring makes either a config change.
- **`num_history_runs` size.** Start ≈3–5 recent turns; tune against real multi-turn
  exchanges (cost vs. context retention). Compression (long-session windowing) is not
  needed at this scope.
- **Grounding strictness vs. helpfulness.** How hard to forbid the model from
  supplementing retrieved recipes with general cooking common-sense — calibrate the
  instruction wording at §6 so it stays grounded without becoming unhelpfully terse.
- **App construction needs config.** Making `build_app()` require `Settings` means the
  importable `app` now depends on a complete env + reachable DB. Confirm the unit
  tier's injected-`Settings` approach (or a lazy/factory construction) keeps `pytest`
  and `uvicorn` import-time behavior clean.
