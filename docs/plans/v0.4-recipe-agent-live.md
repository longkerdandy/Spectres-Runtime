# v0.4 — Recipe Agent Goes Live (Grounded Conversational RAG)

**Status:** planning draft. Architecture and rationale live in the design doc
[`docs/design/recipe-agent.md`](../design/recipe-agent.md); this is the v0.4 work
breakdown. The prior data slice is
[`v0.3-recipe-ingestion.md`](./v0.3-recipe-ingestion.md). Runtime-wide concepts
(AgentOS, Models, Postgres + pgvector) are defined in [`Agents.md`](../../Agents.md).

**Goal:** Give the v0.3 data layer its **first conversational behavior**. v0.4
replaces the placeholder model with a **live domestic chat model** (hosted, OpenAI-compatible),
attaches the v0.3 **knowledge base** so the agent retrieves recipes through Agno's
native agentic RAG (`search_knowledge_base`), wires a **`PostgresDb`** for
**multi-turn sessions** (conversation history + context), and gives the agent
**grounding instructions** — so it answers Chinese recipe questions, in context,
grounded in *retrieved* recipes. This is the first time the runtime issues **real
LLM traffic**.

Where v0.3 was *"data-complete but conversation-empty"*, v0.4 is
**"conversation-complete but journey-empty"**: the agent talks, remembers the
current conversation, and stays grounded — but it only answers **ad-hoc recipe
questions**. The structured **meal-planning journeys** (design §6) and **preference
learning** (User Memory) land in v0.5; **all user-profile / household
personalization** (design §4) is a **separate module**, deferred beyond v0.5 (see
Out of Scope).

**Definition of Done:**

- A live chat model is configured and reachable (§1–§2): a `CHAT_API_KEY` is wired
  through `Settings`, and `build_recipe_agent()` carries a real
  `agno.models.openai.OpenAILike` instead of the placeholder.
- The agent retrieves over the v0.3 corpus (§3–§4): with `knowledge` attached, Agno
  enables `search_knowledge_base`; a Chinese recipe question returns a **non-empty,
  grounded** answer that reflects a retrieved recipe — not the model's parametric
  memory.
- **Multi-turn works** (§3, §6): with `db` attached and `add_history_to_context=True`,
  a follow-up that refers to a previous turn (e.g. 「那它要炒几分钟？」) resolves using
  conversation history; the session persists across runs in Postgres.
- The full path is exercised through **AgentOS's own run endpoint** (§5–§6):
  `POST /agents/recipe/runs` returns a grounded answer; the session surfaces on the
  AgentOS Sessions endpoints.
- Telemetry stays **off** end to end (§5): no anonymous run data leaves the process.
- Gates stay green on the **two-tier split** from v0.3 §10 (§7): unit tests assert
  agent **composition shape** with **no live model call** (coverage ≥ 80%); the live
  answer + multi-turn checks are `integration`, opt-in, and excluded from default CI.

## 1. Chat Model Provider (hosted, OpenAI-compatible domestic LLM)

The chat model is a **hosted, OpenAI-compatible domestic provider**, per
[`Agents.md` → Models](../../Agents.md) (the Runtime leans domestic providers to keep
inference in-region and cheap). The exact provider and tier are **config-driven**,
not wired into code; v0.4 started with a mid-tier model and later validated a
faster channel after the first choice proved unsuitable.

- **Chat model ≠ embedder.** The chat model only generates text; RAG still embeds
  queries with the v0.3 embedder. The two are **separate model ids, separate
  endpoints, separate keys** — do not conflate `CHAT_*` and `EMBEDDER_*`.
- **Provider-agnostic wiring.** The chat model is consumed through Agno's generic
  `OpenAILike` class (`agno.models.openai.OpenAILike`), not a vendor-specific wrapper.
  The model id, base URL, and key all come from config, so swapping to a different
  tier or a different provider is a **config change**, not a code change.
- **Base URL.** No code default — the base URL is a required `Settings` field read from
  `.env`. Set it to whatever endpoint your chosen provider exposes (OpenAI-compatible).
- **Model id is config, not a data contract.** Unlike the embedder (whose id pins the
  stored vector space — v0.3 §1), the chat model id can change freely between runs; it
  governs only generation quality/cost, not stored data. When the household profile +
  health constraints land (v0.5+), a stronger tier with deeper reasoning is expected.

**Deliverables:**

- [x] Generic `OpenAILike` chat model constructed from config, provider-agnostic;
  documented as distinct from the embedder.

## 2. Configuration & Dependencies

Extend the v0.3 `Settings` with the chat-model fields, mirroring the embedder block —
**no defaults, every value from the process env or the git-ignored `.env`** (the v0.3
rule, with no exceptions in this slice).

- **New `Settings` fields:** `chat_model: str`, `chat_base_url: str`,
  `chat_api_key: SecretStr` (all required, no default). A `build_chat_model() -> OpenAILike`
  method mirrors `build_embedder()`: it unwraps the secret and constructs the generic
  `OpenAILike` client. **No `temperature` knob in this version** — generation uses the provider
  default; a configurable temperature can be added later if real usage needs it.
- **`.env.example`** documents the new keys (`CHAT_BASE_URL`, `CHAT_MODEL`, `CHAT_API_KEY`)
  alongside the existing `DATABASE_*` / `EMBEDDER_*` keys.
- **Dependency:** no new third-party package — `OpenAILike` ships with `agno`. (Confirm
  `uv.lock` already resolves it; otherwise `uv add` is not needed.)
- **Telemetry is a code-level concern only (see §5):** it is disabled via `telemetry=False`
  in code on both `AgentOS` and the `Agent` — **no `AGNO_TELEMETRY` env var / `.env` entry**
  is relied on (the `.env`-loaded settings do not reach Agno's `os.getenv` anyway).

**Deliverables:**

- [x] `Settings` carries `chat_*` fields + `build_chat_model()`; `.env.example`
  updated; config unit tests cover load + mapping (no network).

## 3. Agent Composition: model + knowledge + db + multi-turn

Turn `build_recipe_agent()` from a placeholder shell into the real composition from
[design §5](../design/recipe-agent.md#5-agent-composition--rag) — *without* the
personalization legs (deferred). The factory takes `Settings`, and each part is an
**injection seam** (`model` / `knowledge` / `db` default to objects built from
`settings`, but tests pass hermetic doubles) — so unit tests need no monkeypatching.

- **`model`** — `settings.build_chat_model()` (the real `MoonShot`). The
  `_PlaceholderModel` is **removed**: the real model constructs offline (no network
  until a run), so it satisfies AgentOS registration without a stand-in.
- **`knowledge`** — `build_knowledge(settings)` (v0.3). Attaching `knowledge` makes
  Agno enable the built-in **`search_knowledge_base`** tool by default — the agent
  queries the corpus **on demand** (agentic RAG), the dataset is **not** auto-injected
  (design §5).
- **`db`** — `build_db(settings)` (the shared `PostgresDb`, v0.3). Drives **session
  persistence** and conversation history.
- **`instructions`** — `settings.recipe_agent.instructions` (env-driven; see §4).
- **Multi-turn / context (confirmed scope):** `add_history_to_context=True` +
  `num_history_runs=settings.recipe_agent.num_history_runs` (env `RECIPE_AGENT_NUM_HISTORY_RUNS`,
  start N≈3–5) so each run sees the recent turns; `session_id` is supplied per request
  by AgentOS. History is **durable** in Postgres, so a session survives restarts and
  spans requests.
- **`telemetry=False`** on the `Agent` itself (not only on AgentOS — see §5).
- **Still unwired (deferred):** `dependencies` (household injection — §4), profile
  `tools`, and `learning`/User Memory. The factory establishes *where* they attach
  without wiring them.

**Deliverables:**

- [x] `build_recipe_agent(settings, *, model, knowledge, db)` wires real `model` +
  `knowledge` + `db` + `instructions` + multi-turn history; placeholder removed;
  `telemetry=False`; parts injectable for hermetic tests.

## 4. Grounding Instructions & RAG Behavior

The agent is **thin** — its behavior is the model plus `instructions` plus
search-as-tool. The instruction text is **sourced from config**
(`RECIPE_AGENT_INSTRUCTIONS`, read into `Settings.recipe_agent.instructions`) — env-driven
now so it ships with a sane default, and **UI-managed later** without an Agent-shape
change (Agno also accepts a `Callable` for `instructions`, resolved per-run, when a
prompt store is introduced). v0.4 ships the minimal grounding instruction set from
[design §1](../design/recipe-agent.md#1-facts-vs-capability) (facts vs. capability):

- **Ground in retrieved recipes, never parametric memory.** Answer from what
  `search_knowledge_base` returns; if nothing relevant is retrieved, **say so** rather
  than inventing a recipe. The model *processes* retrieved recipes (adapts, explains),
  it does not *recall* them.
- **Reply in the user's language**, recipe-domain persona; cite the dish by name when
  grounding an answer so the grounding is visible/auditable. The citable name is the
  canonical `Recipe.name`, carried in the stored vector **metadata** (sink `_metadata`)
  so it does not depend on the body's title heading.
- **Multi-dish / multi-day menus are in scope.** The agent may compose a full meal
  (several dishes plus a soup) or a multi-day menu in a single answer, **each dish
  grounded in a retrieved recipe** — this is plain grounded generation, not a stored
  plan object.
- **Rely on Agno's native agentic search** (search-as-tool, on by default with
  `knowledge` attached) — **no custom retrieval store**, no `RecipeSource`, no
  whole-dataset injection (design §2.1 / §5).
- **Scope guard (capability honesty, not a single-dish limit):** the agent has **no
  saved, editable meal-plan object** (stateful plan editing → v0.5) and **no household
  / profile data** (`dependencies` injection — separate module). If asked to revise a
  previously saved plan, or to tailor meals to specific people's tastes / portions /
  restrictions, it **says that is not available yet** and offers what it can do now —
  rather than faking a capability it lacks.

**Deliverables:**

- [ ] Grounding instructions shipped as the `RECIPE_AGENT_INSTRUCTIONS` default
  (grounded + dish citation + multi-dish/multi-day + capability-honesty degradation,
  user's language); answers ground in retrieved recipes and degrade to "not found" /
  "not available yet" instead of hallucinating. *(Instruction text shipped; the
  behavioral half is proven at §6.)*

## 5. AgentOS Wiring (config-driven construction; telemetry off)

AgentOS auto-generates its whole HTTP surface from what the agent declares — **we
write no endpoints**. "Integration depth" is therefore a function of which agent
fields §3 attaches. The routes already existed in v0.2/v0.3 (e.g. `POST
/agents/recipe/runs`), but hit the placeholder and raised `NotImplementedError`;
attaching a real `model` + `knowledge` + `db` makes them **functional**.

- **App construction is a factory, not an import-time singleton.** `build_app(agents)`
  takes its agents by **injection** (pure, no I/O of its own); `app_factory()` is the
  production wiring that reads `Settings` and builds the real agent
  (`build_app([build_recipe_agent(settings)])`). Importing the module therefore has
  **no side effects** — no DB, no provider SDK, no credentials — which keeps the unit
  tier hermetic (§7) and removes the old "import requires config + reachable DB"
  coupling. Run with `uvicorn spectres_runtime.app:app_factory --factory`. Construction
  stays network-free (model/DB only touched on a run).
- **Endpoint groups that go live in v0.4:** **Runs** (real LLM + agentic RAG,
  streaming + sync), **Sessions** (durable multi-turn, scoped by `user_id` /
  `session_id`), **Knowledge** (read/search over the 357-recipe corpus), **Traces /
  Metrics** (per-run token usage). Still **off / deferred:** **Memory**
  (no User Memory yet → v0.5), **Auth/JWT** (`authorization=True` left off —
  single-household, deferred module), **Evals**, **HITL / scheduling / background
  jobs**, **Teams / Workflows**, and the **os.agno.com control-plane UI** (opt-in,
  browser→runtime; v0.4 verifies via local HTTP / `/docs` only, never connects it).
- **Privacy posture (kept explicit):** the runtime is **self-hosted**; all session /
  knowledge / trace data lives only in the project's Postgres ("zero transmission" —
  nothing sent to Agno). The **one** default phone-home is Agno's anonymous telemetry
  to `api.agno.com` (model-provider / feature-flag metadata only — never prompts,
  responses, user data, or keys). It is disabled **in code** for defence in depth:
  `telemetry=False` on **both** `AgentOS` (already set) **and** the `Agent` (§3). No
  `AGNO_TELEMETRY` env var / `.env` entry is relied on — the code flags are the single
  source of truth (and Agno only reads `AGNO_TELEMETRY` for the `Agent` leg via
  `os.getenv`, which the `.env`-loaded settings do not populate anyway).
- **Single built-in agent.** `app_factory` registers one agent; `AgentFactory` /
  directory-driven construction remains deferred. The AgentOS `version` (OpenAPI
  `info.version`) is sourced from the package metadata (`pyproject` `version`, now
  `0.4.0`) — one version to maintain, no hand-synced literal.

**Deliverables:**

- [ ] `app_factory()` constructs the live agent from `Settings` via `build_app`;
  Runs/Sessions/Knowledge go live; telemetry off on AgentOS + Agent (code-level).
  (Version is `0.4.0`, already sourced from package metadata.)

## 6. Conversational Retrieval Verification

Prove the loop end to end **through AgentOS's run endpoint**, not just `agent.run()` —
so "AgentOS integration is functional" is actually demonstrated. These checks invoke
the live LLM and embedder, so they are `integration`-tier (opt-in, token-spending).

- **Grounded single-turn (via HTTP):** `POST /agents/recipe/runs` with
  「番茄炒蛋怎么做」 returns a **non-empty** answer that reflects the retrieved dish
  (e.g. mentions egg/tomato steps from `howtocook/vegetable_dish/西红柿炒鸡蛋`). The
  bar is **grounded + relevant**, not an exact string (LLM output is non-deterministic).
- **Multi-turn / context:** within one `session_id`, a first turn asks about a dish,
  a second turn refers back (「那它要炒几分钟？」 / 「换成不放葱呢？」) and the answer
  uses the prior turn — proving `add_history_to_context` works and the session
  persists. Assert the session is retrievable via the Sessions endpoint.
- **Skip-if-unconfigured:** like v0.3's integration checks, skip (not fail) when
  `CHAT_API_KEY` / DB are absent, so the tier is a no-op without credentials.

**Execution note:** Rather than a checked-in integration test (which would cost tokens
on every CI run and is deferred by the user's request), verification is performed
manually via a one-shot script (`/tmp/verify_v6.py`) that exercises the four checks
above against the live chat model. The script is disposable; its output
confirms the agent behavior, after which §5 and §6 deliverables are considered proven.

**Deliverables:**

- [ ] Manual verification drives `POST /agents/recipe/runs` for a grounded Chinese
  answer **and** a 2-turn context-carrying exchange over one session.

## 7. Testing & Quality

Keep the **two-tier split** from v0.3 §10; v0.4 adds the first **chat-token** cost to
the integration tier. The app is a **pure factory** (`build_app(agents)` + a
side-effect-free import), so the unit tier needs no monkeypatching to stay hermetic.

- **Unit tier (default CI, pure).** No DB, no network, **no live model call**. Built on
  **dependency injection**, not patching: `build_recipe_agent(settings, *, model,
  knowledge, db)` and `build_app(agents)` take hermetic doubles. Shared in
  `tests/conftest.py`: `make_settings()` (the `Settings` double, one source of truth),
  `ScriptedModel` (a run-capable `Model` that returns a canned response, bypassing the
  provider pipeline — the modern equivalent of pydantic-ai's `TestModel`), and
  `FakeKnowledge` (a `KnowledgeProtocol` double that connects to nothing). Covers:
  `Settings` chat/agent-field load/mapping; `build_chat_model()` returns a `MoonShot`
  mapped from config (constructed, never invoked); `build_recipe_agent` **shape** — real
  `MoonShot` attached, injected `knowledge` + `db` present, `instructions` from config,
  `add_history_to_context` set, `num_history_runs` from config, `telemetry is False`; a
  **scripted run** (`ScriptedModel` + in-memory `SqliteDb`) proving the wiring is
  invocable offline; and the AgentOS surface (`/health`, `/agents`) via `TestClient`
  over `build_app([...])`.
- **Integration tier (`@pytest.mark.integration`, opt-in).** The §6 live HTTP run +
  multi-turn checks against the real chat API + pgvector.
  Excluded from the default `pytest` run and the coverage gate; skipped without
  credentials. **Token cost (new):** chat completions in addition to query embeddings.
- **Coverage honesty.** Integration-only glue (`app_factory` production wiring, the
  `recipe-ingest` entry from v0.3) is scoped out of the unit denominator (`pragma: no
  cover`), per v0.3 §10 — the gate measures unit-testable code only.
- **CI stays token-free.** `ci.yml` keeps running bare `pytest` (`-m "not
  integration"`); no `CHAT_API_KEY`, embedder key, or DB in CI. The live tier is local
  / optional secret-gated, never on the default push path.

**Deliverables:**

- [x] Unit tier green and pure (no live model), DI-based (no monkeypatching), coverage
  ≥ 80%.
- [ ] Integration tier covers the live grounded + multi-turn run, excluded from default
  CI, runnable locally.

## Explicitly Out of Scope for v0.4

v0.4 is the **"agent goes live as grounded conversational Q&A"** slice only.

- **Stateful meal-plan journeys** (design §6) — a **saved, editable plan object** and
  **conversational plan adjustment** over it ("swap Wednesday", "add a soup to
  yesterday's plan"). → v0.5. *(Note: composing a multi-dish / multi-day menu in a
  single grounded answer is **in scope** at §4 — what's deferred is persisting it as a
  referenceable, editable object, not generating menu text.)*
- **User Memory / preference learning** (Agno Learning Machine — "likes spicy",
  "found the braised pork too sweet"). Note this is **distinct** from Profile
  Management. → v0.5.
- **All user-profile / Profile Management work** (design §4) — `User` / `Household`,
  membership, **merged household constraints**, `dependencies` injection, member-detail
  / record-measurement **tools**, dietary restrictions / conditions / metrics. This is
  a **separate, independent module**, deferred beyond v0.5 (per project requirement).
- **Nutrition reasoning** / health-target matching (needs a nutrition-facts source).
- **Auth / JWT / RBAC** (`authorization=True`), **os.agno.com control-plane** linkage,
  **Teams / Workflows**, **`AgentFactory`** / directory-driven construction.
- **Retrieval upgrades** still deferred from v0.3: ANN index, Chinese FTS /
  `SearchType.hybrid`, reranking — add only if real v0.4 usage shows weak precision.
- **Media / object store** (image serving), **containerization / CD**, **Alembic
  migrations**, **Edge / Client**.

## Resolved Questions

- **Chat provider: config-driven, provider-agnostic.** The first evaluated channel
  proved unsuitable (coding-agent-only restriction); a second channel was validated
  manually and selected for speed and stability. The wiring remains provider-agnostic
  via Agno's generic `OpenAILike`; swapping to a stronger tier (for v0.5+ household
  profile / health constraints) or another provider is a config change.
- **Multi-turn included.** Confirmed (requirement): the agent carries **conversation
  context + durable history** via `db` + `add_history_to_context=True` +
  `num_history_runs`, with `session_id` managed per request by AgentOS. (Stateful
  *meal-plan* editing — a richer object than chat history — is still v0.5.)
- **AgentOS integration depth.** Static single built-in agent; we write no endpoints.
  Attaching `model`/`knowledge`/`db` makes **Runs / Sessions / Knowledge / Traces**
  functional; Memory / Auth / Evals / HITL / Teams / control-plane UI stay off. Verified
  via local HTTP (`TestClient` / `/docs`), never by connecting os.agno.com.
- **Self-hosted, no cloud upload.** Runtime data stays in the project's Postgres;
  AgentOS sends nothing to Agno. The one default phone-home is anonymous telemetry to
  `api.agno.com` (provider/feature metadata, no content) — disabled in code on AgentOS +
  Agent (`telemetry=False`); no env-var / `.env` opt-out is relied on.
- **Chat model id is not a data contract.** It can change run-to-run (unlike the
  embedder id, which pins the stored vector space); only generation quality/cost
  depend on it. A mid-tier model was selected for v0.4; a stronger tier is expected
  when household profile / health constraints land (v0.5+).

## Open Questions

- **`num_history_runs` size.** Start ≈3–5 recent turns; tune against real multi-turn
  exchanges (cost vs. context retention). Compression (long-session windowing) is not
  needed at this scope.
- **Grounding strictness vs. helpfulness.** How hard to forbid the model from
  supplementing retrieved recipes with general cooking common-sense — calibrate the
  instruction wording at §6 so it stays grounded without becoming unhelpfully terse.
- **Model tier validation.** The mid-tier model's quality for grounded recipe Q&A and
  multi-turn context retention is confirmed at §6 (manual verification). Whether it
  remains sufficient when multi-dish menu composition becomes more complex, and when
  a stronger reasoning tier is needed, is assessed at v0.5 when household profile
  constraints land. Provider-agnostic wiring makes any swap a config change.
- **App construction needs config.** *(Resolved in §3/§5/§7.)* Rather than an
  import-time `app = build_app(settings)` singleton, the app is now a **pure factory**:
  `build_app(agents)` takes injected agents, `app_factory()` does the production
  `Settings`-driven wiring, and importing the module has no side effects. `pytest`
  stays hermetic without a DB/env; `uvicorn` uses `--factory`.
