# v0.4.1 开发计划：Recipe Agent Tool-Driven 重构

**Status:** 讨论稿（Draft）
**目标:** 将单次对话 input token 从 ~60,000 降至 ~1,500，同时保留菜谱的可溯源性与可执行性。
**交付版本:** v0.4.1（patch release，向后兼容 ingestion 与 schema）

---

## 1. 问题陈述

当前 Recipe Agent 采用 Agno 原生 Agentic RAG：`knowledge` 对象挂载在 Agent 上，`search_knowledge=True` 开启自动注入。实测一次 "4菜1汤" 请求产生 **60,066 input tokens**，原因：

- Agent 自主触发 **3 轮** `search_knowledge_base`
- 每轮返回 **10 个文档**
- 每个文档注入完整 Markdown 菜谱（平均 ~2,000 token）

**核心矛盾：** 系统把"全书"塞给 Agent，但"搭配菜单"这个决策任务只需要"目录"。

---

## 2. 新架构：Tool-Driven 精确检索

**范式转换：** 从"系统被动注入检索结果"转为"Agent 按需调用工具获取信息"。

```
┌─────────────────────────────────────────────────────────────┐
│  用户: "帮我搭配 4菜1汤"                                      │
│                                                             │
│  Agent 思考: "需要 1汤 + 2荤 + 2素"                         │
│    ├─ 调用 search_recipes(query="清淡汤", category="soup")   │
│    ├─ 调用 search_recipes(query="下饭菜", category="meat_dish")│
│    └─ 调用 search_recipes(query="素菜", category="vegetable_dish")│
│                                                             │
│  Agent 组合 11 个精简候选 → 输出菜单                          │
│                                                             │
│  用户: "西红柿鸡蛋汤怎么做？"                                 │
│    └─ 调用 get_recipe_detail(recipe_id="howtocook/西红柿鸡蛋汤")│
│         → 返回完整 Markdown（~1,500 token）                   │
│                                                             │
│  Agent 基于真实步骤回答                                       │
└─────────────────────────────────────────────────────────────┘
```

**关键原则：**
- `search_recipes` 返回 **metadata 层**（目录信息），不返回 steps
- `get_recipe_detail` 返回 **完整内容层**（含 steps），按需精确查询
- Agent 通过 instructions 理解何时用哪个 tool

## 3. 接口定义

### 3.1 Tool: `search_recipes`

**职责：** 语义搜索菜谱目录，返回精简信息供 Agent 做搭配决策。支持向量语义 + PgVector JSONB 精确过滤。

```python
from agno.tools import tool

@tool(name="search_recipes")
def search_recipes(
    query: str,
    category: str | None = None,
    must_include: list[str] | None = None,
    must_exclude: list[str] | None = None,
    limit: int = 4,
) -> str:
    """
    Search the recipe catalog by semantic meaning with optional ingredient filters.

    Use this tool to discover candidate dishes when planning a menu.
    Returns lightweight metadata only — no cooking steps.

    Args:
        query: What you are looking for (e.g., "清淡汤", "下饭菜", "快手素菜").
               Describe the desired flavor, style, or occasion in Chinese.
        category: Optional filter. One of: "soup", "meat_dish", "vegetable_dish",
                  "staple", "breakfast", "dessert", "condiment".
        must_include: Optional list of ingredients that the recipe MUST contain
                      (e.g., ["西红柿", "鸡蛋"]). All listed ingredients must be present.
        must_exclude: Optional list of ingredients or tags that the recipe MUST NOT contain
                      (e.g., ["辣椒", "花椒", "五花肉"]). Any match excludes the recipe.
        limit: Maximum number of candidates to return (default 4, max 8).

    Returns:
        JSON string with a list of recipe summaries. Each entry has:
        - recipe_id: stable identifier (e.g., "howtocook/西红柿鸡蛋汤")
        - name: display name
        - category: list of categories
        - difficulty: 1-5
        - description: one-sentence summary
        - ingredients: list of ingredient names
    """
```

**内部实现要点：**

**数据源：** 直接查询 `public.recipes` PgVector 表（绕过 Agno 的 `knowledge.search()` filter abstraction，因为 Agno 的 `filters` 参数不支持 JSONB 数组的 `@>` / `NOT @>` 条件）。

**查询构造：**
```python
# 1. 用共享 embedder 生成 query vector
embedding = settings.build_embedder().get_embedding(query)

# 2. 原生 SQL：向量相似度排序 + JSONB filter
sql = """
SELECT
    name AS recipe_id,
    metadata->>'name' AS name,
    metadata->'category' AS category,
    metadata->>'difficulty' AS difficulty,
    metadata->>'description' AS description,
    metadata->'ingredients' AS ingredients,
    vector <=> %s::vector AS distance
FROM recipes
WHERE 1=1
"""
params = [embedding]

# category filter (JSONB array contains)
if category:
    sql += " AND metadata->'category' @> %s::jsonb"
    params.append(json.dumps([category]))

# must_include: ingredients array must contain all specified names
if must_include:
    for ing in must_include:
        sql += " AND metadata->'ingredients' @> %s::jsonb"
        params.append(json.dumps([{"name": ing}]))

# must_exclude: ingredients array must NOT contain any specified name
if must_exclude:
    for ing in must_exclude:
        sql += " AND NOT metadata->'ingredients' @> %s::jsonb"
        params.append(json.dumps([{"name": ing}]))

sql += " ORDER BY vector <=> %s::vector LIMIT %s"
params.extend([embedding, limit])
```

**返回处理：**
- 从查询结果提取 metadata 字段，不返回 `text_content` 全文
- `description` 优先取 metadata 中的预存字段；如无则运行时从 content 截断前 120 字
- 序列化为 JSON string 返回（Agno tool 返回 str）

**Token 预算：**
- 4 个结果 × 每结果 ~80 token（metadata）= **~320 token**

### 3.2 Tool: `get_recipe_detail`

**职责：** 按稳定 ID 精确获取单道菜谱的完整内容。

```python
@tool(name="get_recipe_detail")
def get_recipe_detail(recipe_id: str) -> str:
    """
    Retrieve the full recipe (ingredients with quantities and step-by-step instructions).

    Use this tool ONLY when the user asks "怎么做", "详细做法", "步骤", or asks
    for a specific dish by name after seeing it on a menu.

    Args:
        recipe_id: The stable recipe identifier returned by search_recipes
                   (e.g., "howtocook/西红柿鸡蛋汤").

    Returns:
        The complete recipe in Markdown format, including ingredients, quantities,
        and numbered steps. If not found, returns a clear "未找到" message.
    """
```

**内部实现要点：**
- 直接查 `public.recipes` PgVector 表：`SELECT text_content FROM recipes WHERE name = %s`
- 或查 Agno `contents_db` 表：`SELECT content FROM agno_knowledge WHERE name = %s`
- 需要调研确认哪个表/列存储了完整的 `text_content`
- 返回完整 Markdown（上限 3,000 字，防极端情况截断）

**Token 预算：**
- 1 个结果 × ~1,500 token = **~1,500 token**

### 4.1 `build_recipe_agent` 改动

def build_recipe_agent(
    settings: Settings,
    *,
    model: Model | None = None,

### 4.2 Instructions 更新

新增/替换为：

```markdown
你是 Spectres 个人助手的 Recipe Agent。

**你的工作方式：**
- 你拥有两个工具：`search_recipes`（搜索菜谱目录）和 `get_recipe_detail`（获取完整做法）。

  - 使用 `search_recipes` 返回的 recipe_id 精确查询。
  - 基于返回的完整 Markdown 回答，不要凭记忆编造步骤或用量。
- **不要在没有工具结果的情况下编造菜谱步骤或用量。**

**回答规则：**
- 所有推荐的菜必须能追溯到具体的 recipe_id。
- 使用用户的语言（中文）回答。
```

## 7. 风险与回滚方案

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| `get_recipe_detail` 查不到内容（schema 未知） | 用户追问时 Agent 无法回答 | #1 先行调研；如 contents_db 不可靠，直接 SQL 查 `recipes` 表 |
| Agent 不调用 tool，直接 hallucinate 步骤 | 回答不可靠 | Instructions 明确禁止；integration test 验证 tool 调用痕迹 |
| `search_recipes` 的 description 截断导致语义丢失 | 检索质量下降 | 优先调研 metadata 是否已有 `description`；如没有，#9 预生成 summary |
| 多轮对话中重复调用 `get_recipe_detail` | Token 仍可能累积 | 短期可接受（追问 1-2 道菜）；长期可加 conversation-level cache |
| Tool 调用的延迟 | 响应时间增加 | Tool 在本地执行，无网络；延迟来自 DB 查询，可忽略 |
| PgVector JSONB filter 语法与 Agno schema 不兼容 | `must_include` / `must_exclude` 过滤失效 | #1 先行验证 JSONB 查询语法；如 Agno 的 metadata 结构不同，调整 SQL |
| `must_include` / `must_exclude` 过滤后结果不足 `limit` | 返回空列表或少量结果 | 预期行为：告诉用户"没有同时满足这些条件的菜"；或放宽 limit 后二次筛选 |

**回滚方案：**
- 代码层面：`build_recipe_agent` 保留 knowledge 参数作为 fallback（注释掉而非删除）
- 配置层面：通过 env flag `RECIPE_AGENT_USE_TOOLS=true/false` 切换新旧模式
- 数据层面：ingestion 和 schema 零改动，回滚只需改 Agent 构造

## 9. 已通过代码确认的事实

以下问题通过阅读现有代码和目录结构已得到答案，无需再确认：

### 9.1 `get_recipe_detail` 的数据源

**答案：直接查 `public.recipes` 表的 `text_content` 列。**

`recipe_agent/ingestion/sink.py:64-68` 明确显示：
```python
self._knowledge.insert(
    name=recipe.id,              # 即 PgVector 表的 name 列
    text_content=recipe.content,  # 完整 Markdown 存入 vector_db
    metadata=self._metadata(recipe),
    upsert=True,
)
```

Agno 的 `Knowledge.insert()` 会把 `text_content` 同时写入 `recipes` 表（PgVector）和 `agno_knowledge` 表（contents_db）。`get_recipe_detail` 直接 SQL 查 `recipes` 表即可：
```sql
SELECT text_content FROM recipes WHERE name = %s
```

### 9.2 `recipes` 表的 schema

**`name` 列格式：** `howtocook/<category>/<dish_name>`

来源：`howtocook.py:96`：`id=f"{self.name}/{ref.removesuffix('.md')}"`
- `self.name` = `"howtocook"`
- `ref` = `"aquatic/咖喱炒蟹.md"`
- `id` = `"howtocook/aquatic/咖喱炒蟹"`

**`metadata` 中的 `ingredients` 结构：** `[{"name": "xxx", "optional": false}]`

来源：`sink.py:96`：
```python
"ingredients": [{"name": i.name, "optional": i.optional} for i in recipe.ingredients]
```

### 9.3 Category 取值集合

**来源：** `datasets/howtocook/dishes/` 的子目录名：

| Category | 中文含义 | 菜谱数（估算） |
|---|---|---|
| aquatic | 水产 | ~30 |
| breakfast | 早餐 | ~15 |
| condiment | 酱料/调味品 | ~20 |
| dessert | 甜品 | ~25 |
| drink | 饮品 | ~15 |
| meat_dish | 荤菜 | ~80 |
| semi-finished | 半成品 | ~10 |
| soup | 汤 | ~40 |
| staple | 主食 | ~50 |
| template | 模板（非菜谱） | ~5 |
| vegetable_dish | 素菜 | ~60 |

**`template` 不是真实菜谱**，`search_recipes` 的 SQL 应该自动排除：
```sql
AND NOT metadata->'category' @> '["template"]'
```

| # | 问题 | 决策 |

| 2 | AgentOS knowledge 注册 | **删除**（详见 §10.1） |

### 10.1 删除 AgentOS knowledge 注册的影响分析

当前 `app.py`：
```python
knowledge = build_recipe_knowledge(settings)
agent = build_recipe_agent(settings, knowledge=knowledge)
return build_app([agent], knowledge=[knowledge])
```

**删除 `knowledge=[knowledge]` 的影响：**

| 受影响方 | 影响 | 处理 |
|---|---|---|
| **AgentOS 控制平面 `/knowledge/*` 端点** | Recipe Knowledge 不再出现在控制平面列表中 | 可接受。v0.4.1 不依赖 AgentOS 管理 knowledge |
| **`build_recipe_agent`** | 不再需要 `knowledge` 参数 | 删除参数，简化签名 |
| **`recipe-ingest` CLI** | **不受影响**。Ingestion 仍然调用 `build_recipe_knowledge` 写入数据 | 保持现状 |
| **测试 `FakeKnowledge`** | 不再需要（Agent 不依赖 knowledge 对象） | 删除或简化 |

**关键澄清：**
- `build_recipe_knowledge()` 函数**不删除**——它仍然被 `recipe-ingest` CLI 使用
- 只是 `build_recipe_agent()` 不再接收 `knowledge` 参数，AgentOS 注册时不再传 `knowledge=[...]`
- `search_recipes` tool 内部用原生 SQL + 共享 embedder，不依赖 Agno `Knowledge` 对象

---

### 10.2 删除后仍需保留的代码

以下代码**保留**，因为它们不依赖 AgentOS 注册：

1. **`build_recipe_knowledge()`**（`recipe_agent/knowledge.py`）— 被 `recipe-ingest` CLI 使用
2. **`RecipeSink`**（`recipe_agent/ingestion/sink.py`）— 写入逻辑不变
3. **`PgVector` 配置**（`storage/knowledge.py`）— 向量表结构不变
4. **`recipes` 表** — 数据不动，只是查询方式从 Agno API 改为原生 SQL
