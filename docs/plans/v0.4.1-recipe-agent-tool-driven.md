# v0.4.1 Development Plan: Tool-Driven Recipe Agent Refactor

**Status:** Draft
**Goal:** Reduce single-conversation input tokens from ~60,000 to ~1,500 while preserving recipe traceability and executability.
**Target Version:** v0.4.1 (patch release, backward-compatible ingestion & schema)

---

## 1. Problem Statement

The current Recipe Agent uses Agno's native Agentic RAG: a `knowledge` object is mounted on the Agent with `search_knowledge=True`, which automatically injects retrieved documents into the prompt. A single "4 dishes + 1 soup" request generates **60,066 input tokens** because:

- The Agent autonomously triggers **3 rounds** of `search_knowledge_base`
- Each round returns **10 documents**
- Each document injects the full Markdown recipe (avg. ~2,000 tokens)

**Core contradiction:** The system feeds the Agent entire "books", but the "menu planning" decision task only needs a "catalog".

---

## 2. New Architecture: Tool-Driven Precise Retrieval

**Paradigm shift:** From "system passively injects retrieval results" to "Agent actively calls tools to fetch information on demand".

```
┌─────────────────────────────────────────────────────────────┐
│  User: "Plan a 4-dish + 1-soup menu"                        │
│                                                             │
│  Agent reasons: "Need 1 soup + 2 meat + 2 vegetable"        │
│    ├─ Call search_recipes(query="light soup", category="soup")│
│    ├─ Call search_recipes(query="savory meat", category="meat_dish")│
│    └─ Call search_recipes(query="vegetable", category="vegetable_dish")│
│                                                             │
│  Agent combines 11 lightweight candidates → outputs menu    │
│                                                             │
│  User: "How do I make tomato egg drop soup?"                │
│    └─ Call get_recipe_detail(recipe_id="howtocook/西红柿鸡蛋汤")│
│         → Returns full Markdown (~1,500 tokens)             │
│                                                             │
│  Agent answers based on real steps                          │
└─────────────────────────────────────────────────────────────┘
```

**Key principles:**
- `search_recipes` returns the **metadata layer** (catalog info), no cooking steps
- `get_recipe_detail` returns the **full content layer** (with steps), fetched precisely on demand
- The Agent understands when to use which tool via instructions

## 3. Development Steps

### Step 1: Data Layer Investigation ✅ COMPLETED

**Goal:** Verify the exact schema of the `recipes` table and confirm data storage layout.

**Findings:**
- `recipes` table columns: `id`, `name` (recipe_id), `content` (full Markdown), `embedding` (vector), `meta_data` (JSONB), plus Agno housekeeping columns (`filters`, `usage`, `content_hash`, `content_id`, `created_at`, `updated_at`)
- `name` format: `howtocook/<category>/<dish_name>` (e.g., `howtocook/aquatic/咖喱炒蟹`)
- `meta_data.ingredients` structure: `[{"name": "xxx", "optional": false}]`
- `meta_data` contains Agno internal fields: `chunk`, `chunk_size`, `linked_to`
- `meta_data` does **NOT** contain a `description` field (confirmed by DB inspection)
- Categories: aquatic, breakfast, condiment, dessert, drink, meat_dish, semi-finished, soup, staple, template, vegetable_dish
- `template` is not a real recipe and must be excluded from search
- `content` holds the full Markdown with steps, quantities, and tips (avg ~724 chars, max ~4,546 chars)
- `agno_knowledge` table (contents_db) mirrors recipe metadata but is **not needed** for tool queries

**Deliverable:** `docs/plans/v0.4.1-task1-data-layer-report.md`

### Step 3: Implement `search_recipes` Tool

**Goal:** Build the catalog search tool with PgVector JSONB filtering.

**Tool signature:**

```python
from agno.tools import tool

@tool(name="search_recipes")
def search_recipes(
    query: str,
    category: str | None = None,
    must_include: list[str] | None = None,
    must_exclude: list[str] | None = None,
    limit: int = 4,
) -> str:
    """
    Search the recipe catalog by semantic meaning with optional ingredient filters.

    Use this tool to discover candidate dishes when planning a menu.
    Returns lightweight metadata only — no cooking steps.

    Args:
        query: What you are looking for (e.g., "light soup", "savory dish", "quick veggie").
               Describe desired flavor, style, or occasion in Chinese.
        category: Optional filter. One of: "soup", "meat_dish", "vegetable_dish",
                  "staple", "breakfast", "dessert", "condiment".
        must_include: Ingredients the recipe MUST contain
                      (e.g., ["tomato", "egg"]). All must be present.
        must_exclude: Ingredients the recipe MUST NOT contain
                      (e.g., ["chili", "peppercorn", "pork belly"]). Any match excludes.
        limit: Maximum candidates (default 4, max 8).

    Returns:
        JSON string with recipe summaries. Each entry:
        - recipe_id: stable ID (e.g., "howtocook/soup/tomato_egg_drop")
        - name: display name
        - category: list
        - difficulty: 1-5
        - description: AI-generated one-sentence description
        - ingredients: list of names
    """
```

**Implementation details:**

File: `src/spectres_runtime/recipe_agent/tools.py`

```python
import json
from typing import Any

import psycopg
from agno.tools import tool

from spectres_runtime.config import Settings


def _build_search_sql(
    embedding: list[float],
    category: str | None,
    must_include: list[str] | None,
    must_exclude: list[str] | None,
    limit: int,
) -> tuple[str, list[Any]]:
    sql = """
    SELECT
        name AS recipe_id,
        meta_data->>'name' AS name,
        meta_data->'category' AS category,
        meta_data->>'difficulty' AS difficulty,
        meta_data->>'description' AS description,
        meta_data->'ingredients' AS ingredients,
        embedding <=> %s::vector AS distance
    FROM recipes
    WHERE 1=1
      AND NOT meta_data->'category' @> '["template"]'::jsonb
    """
    params: list[Any] = [embedding]

    if category:
        sql += " AND meta_data->'category' @> %s::jsonb"
        params.append(json.dumps([category]))

    if must_include:
        for ing in must_include:
            sql += " AND meta_data->'ingredients' @> %s::jsonb"
            params.append(json.dumps([{"name": ing}]))

    if must_exclude:
        for ing in must_exclude:
            sql += " AND NOT meta_data->'ingredients' @> %s::jsonb"
            params.append(json.dumps([{"name": ing}]))

    sql += " ORDER BY embedding <=> %s::vector LIMIT %s"
    params.extend([embedding, limit])

    return sql, params


def build_search_recipes_tool(settings: Settings):
    embedder = settings.build_embedder()
    db_url = settings.database_url.replace("postgresql+psycopg://", "postgresql://", 1)

    @tool(name="search_recipes")
    def search_recipes(
        query: str,
        category: str | None = None,
        must_include: list[str] | None = None,
        must_exclude: list[str] | None = None,
        limit: int = 4,
    ) -> str:
        embedding = embedder.get_embedding(query)
        sql, params = _build_search_sql(embedding, category, must_include, must_exclude, limit)

        with psycopg.connect(db_url) as conn:
            with conn.cursor() as cur:
                cur.execute(sql, params)
                rows = cur.fetchall()

        results = []
        for row in rows:
            results.append({
                "recipe_id": row[0],
                "name": row[1],
                "category": row[2],
                "difficulty": row[3],
                "description": row[4],
                "ingredients": [i["name"] for i in (row[5] or [])],
            })

        return json.dumps(results, ensure_ascii=False)

    return search_recipes
```

**Token budget:** 4 results × ~80 tokens each = **~320 tokens**

**Acceptance Criteria:**
- Returns JSON array with correct metadata fields
- Category filter works (test with "soup" → only soups)
- `must_include` works (test with ["tomato", "egg"] → recipes containing both)
- `must_exclude` works (test with ["chili"] → no spicy recipes)
- Never returns `template` category recipes
- Does not return `content`

**Estimated effort:** 2 hours

### Step 4: Implement `get_recipe_detail` Tool

**Goal:** Build the precise detail retrieval tool.

**Tool signature:**

```python
@tool(name="get_recipe_detail")
def get_recipe_detail(recipe_id: str) -> str:
    """
    Retrieve the full recipe with ingredients, quantities, and step-by-step instructions.

    Use ONLY when the user asks "how to make", "detailed steps", or asks about
    a specific dish by name after seeing it on a menu.

    Args:
        recipe_id: Stable ID from search_recipes (e.g., "howtocook/soup/tomato_egg_drop").

    Returns:
        Complete recipe in Markdown. Includes ingredients with quantities
        and numbered steps. Returns "Recipe not found" if missing.
    """
```

**Implementation:**

```python
def build_get_recipe_detail_tool(settings: Settings):
    db_url = settings.database_url.replace("postgresql+psycopg://", "postgresql://", 1)

    @tool(name="get_recipe_detail")
    def get_recipe_detail(recipe_id: str) -> str:
        with psycopg.connect(db_url) as conn:
            with conn.cursor() as cur:
                cur.execute(
                    "SELECT content FROM recipes WHERE name = %s LIMIT 1",
                    (recipe_id,),
                )
                row = cur.fetchone()

        if not row or not row[0]:
            return f"Recipe '{recipe_id}' not found."

        content = row[0]
        # Safety cap: truncate extremely long recipes
        max_len = 3000
        if len(content) > max_len:
            content = content[:max_len] + "\n\n... (truncated)"

        return content

    return get_recipe_detail
```

**Token budget:** 1 result × ~1,500 tokens = **~1,500 tokens**

**Acceptance Criteria:**
- Returns full Markdown for existing recipe_id
- Returns clear "not found" message for invalid ID
- Truncates content longer than 3,000 chars with ellipsis
- Query completes in < 100ms

**Estimated effort:** 1 hour

### Step 5: Refactor `build_recipe_agent`

**Goal:** Remove knowledge auto-injection, wire tools, embed instructions.

**Changes to `src/spectres_runtime/recipe_agent/agent.py`:**

```python
from spectres_runtime.recipe_agent.tools import (
    build_get_recipe_detail_tool,
    build_search_recipes_tool,
)

# New: embedded instructions (no longer env-driven)
RECIPE_AGENT_INSTRUCTIONS = """\
You are the Recipe Agent for the Spectres personal assistant.

**How you work:**
- You have two tools: `search_recipes` (catalog search) and `get_recipe_detail` (full recipe).
- When the user asks for menu planning, dish recommendations, or "what can I cook",
  **use `search_recipes`**.
  - Analyze the need first (e.g., 1 soup + 2 meat + 2 vegetable).
  - Call `search_recipes` per category (category = soup / meat_dish / vegetable_dish).
  - Combine candidates into a balanced menu. Cite each dish's recipe_id and source.
- When the user asks "how do I make X", "detailed steps", or "what ingredients/quantities",
  **use `get_recipe_detail`**.
  - Use the recipe_id returned by `search_recipes`.
  - Base your answer on the returned Markdown. Do not invent steps or quantities.
- **Never invent recipe steps or quantities without tool results.**

**Rules:**
- Every recommended dish must trace back to a specific recipe_id.
- Reply in the user's language (Chinese).
"""


def build_recipe_agent(
    settings: Settings,
    *,
    model: Model | None = None,
    db: BaseDb | None = None,
) -> Agent:
    """Construct the recipe agent with tool-driven retrieval."""
    return Agent(
        id=RECIPE_AGENT_ID,
        name=RECIPE_AGENT_NAME,
        model=model or settings.recipe_agent.build_chat_model(),
        db=db or build_db(settings),
        # Removed: knowledge, search_knowledge, add_knowledge_to_context
        tools=[
            build_search_recipes_tool(settings),
            build_get_recipe_detail_tool(settings),
        ],
        instructions=RECIPE_AGENT_INSTRUCTIONS,
        add_history_to_context=True,
        num_history_runs=settings.recipe_agent.num_history_runs,
        user_id=DEFAULT_USER_ID,
        telemetry=False,
    )
```

**Acceptance Criteria:**
- Agent constructs without errors
- No `knowledge` parameter in signature
- Tools are registered and callable
- Instructions are embedded, not read from env

**Estimated effort:** 1 hour

## 6. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| AI description generation produces low-quality output | 357 recipes have bad descriptions | Batch-test on 10 recipes first; review outputs before full run |
| Agent doesn't call tools, hallucinates steps | Unreliable answers | Instructions explicitly forbid; integration tests verify tool call traces |
| PgVector JSONB filter syntax mismatch | Filters silently fail | Test each filter type (category, must_include, must_exclude) in isolation |
| `must_include` / `must_exclude` returns no results | Empty menu suggestions | Expected behavior; Agent should inform user no matching recipes exist |
| Tool latency spikes | Slow responses | Tools execute locally (no network); DB queries should be < 100ms with proper indexing |
| AgentOS knowledge endpoints broken | Control plane can't manage recipes | Acceptable trade-off; ingestion CLI still works independently |
