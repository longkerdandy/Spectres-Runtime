# v0.3 — Recipe Ingestion & Knowledge Base

**Status:** planning draft. Architecture and rationale live in the design doc
[`docs/design/recipe-agent.md`](../design/recipe-agent.md); this is the v0.3 work
breakdown. The prior skeleton is
[`v0.2-recipe-agent-skeleton.md`](./v0.2-recipe-agent-skeleton.md). Runtime-wide
concepts (AgentOS, Postgres + pgvector, Models) are defined in
[`Agents.md`](../../Agents.md).

**Goal:** Give the v0.2 skeleton its **first real behavior on the data side**:
turn the behavior-empty stubs into a working ingestion path that fills a
**searchable recipe knowledge base**. v0.3 stands up Postgres + pgvector, wires a
local embedding model (**Qwen3-Embedding-8B**, GPU-accelerated), implements the
**HowToCook** ingester (vendored snapshot → Markdown → `Recipe`), and persists
each recipe through `RecipeSink` into pgvector — then proves the data is
retrievable via Agno's native knowledge search.

Where v0.2 was *"structurally-complete but behavior-empty"*, v0.3 is
**"data-complete but conversation-empty"**: real recipes are parsed, embedded,
stored, and retrievable — but the agent is **still not wired to a live chat
model** and answers no user journeys. Live LLM traffic, the agent's RAG
composition, and **all personalization** stay deferred to v0.4+ (see Out of
Scope). Isolating ingestion gives v0.3 a **deterministic** acceptance bar (N
recipes in, relevant recipe out) with no LLM-evaluation subjectivity.

**Definition of Done:**

- A local GPU inference environment is documented and verified (§1): the 5090 is
  visible inside WSL2, PyTorch runs `sm_120` kernels, and the embedder produces a
  4096-dim vector.
- `docker compose up` brings up Postgres + pgvector; the app connects via a
  configured `DATABASE_URL` (§2–§3).
- `HowToCookIngester.ingest()` yields typed `Recipe` objects parsed from a
  **vendored, pinned** HowToCook snapshot; `RecipeSink.write()` embeds and upserts
  them into pgvector, returning honest `written / skipped / failed` counts and
  **idempotent** on re-run (§5–§7).
- A single ingestion command populates the knowledge base; a retrieval check
  returns a relevant recipe for a Chinese-language query — **no chat LLM
  involved** (§8–§9).
- Gates stay green with a **new test tier split** (§10): pure unit tests (parser,
  mapping) keep the v0.1 bar (no DB / network / GPU, coverage ≥ 80%); DB- and
  embedder-dependent tests are marked `integration` and run opt-in.

## 1. GPU / Local Inference Environment

Qwen3-Embedding-8B (~8B params, 4096-dim output) runs locally, GPU-accelerated.
The dev box is an **RTX 5090 (Blackwell, `sm_120`)** under **WSL2**. This section
makes that environment reproducible and verifiable — it is a prerequisite for
every embedding step below.

- **GPU passthrough into WSL2.** The NVIDIA driver is installed on the *Windows*
  host (never inside WSL); WSL2 reaches the GPU through the host driver. Verify
  with `nvidia-smi` inside WSL listing the RTX 5090. WSL1 has no GPU passthrough.
- **Blackwell needs a CUDA 12.8 (`cu128`) PyTorch build.** Older `torch` wheels
  ship no `sm_120` kernels and fail with *"no kernel image is available"* or
  silently fall back to CPU. Install `torch` from the **cu128** index (pinned via
  `[tool.uv.sources]`, §3). `is_available() == True` is **not** sufficient proof —
  a real CUDA matmul must succeed.
- **Verification gate (documented in `docs/environment.md`):**

  ```bash
  nvidia-smi                                   # RTX 5090 listed
  uv run python - <<'PY'
  import torch
  assert torch.cuda.is_available(), "CUDA not available"
  assert torch.cuda.get_device_capability()[0] >= 12, "not Blackwell-capable torch"
  print(torch.cuda.get_device_name(0))
  print((torch.randn(8, device="cuda") @ torch.randn(8, device="cuda")).item())  # kernel runs
  PY
  ```

- **HF cache on the native filesystem.** Point `HF_HOME` at a WSL-native path
  (e.g. `~/.cache/huggingface`), **never** under `/mnt/c/...` — cross-filesystem
  I/O makes loading the multi-GB weights slow.
- **GPU is dev-only, never required by CI.** Ingestion runs on the dev box with
  the 5090; CI never downloads the 8B weights nor needs a GPU (see §10 for how the
  test tiers enforce this).

**Deliverables:**

- [x] `docs/environment.md` documents WSL2 GPU setup + the cu128 verification
  gate; the gate passes locally on the 5090.

## 2. Infrastructure: Postgres + pgvector

Stand up the single datastore `Agents.md` mandates — Postgres + pgvector, the
source of truth for knowledge vectors (and, later, sessions / memory / profile).

- **`docker/compose.yaml`** runs Agno's official **`agnohq/pgvector:16`** image
  (Postgres 16 with the `vector` extension preinstalled), with a named volume for
  durability and a `pg_isready` health check. This is the only service v0.3 needs.
  Follows Agno's host-port convention (**5532**→5432) so it never clashes with a
  standard local Postgres or sibling projects on 5432.
- **App containerization is *not* in scope.** The Runtime still runs via `uvicorn`
  on the host; only the database is containerized. A full app image / compose
  topology is deferred.
- The `vector` extension is enabled on first boot via a committed init SQL
  (`docker/postgres/init.sql`, mounted into `/docker-entrypoint-initdb.d`) so the
  DB is self-sufficient — independent of Agno's table-creation path, which also
  ensures it.

**Deliverables:**

- [x] `docker compose -f docker/compose.yaml up -d` yields a healthy Postgres +
  pgvector reachable at `postgresql+psycopg://ai:ai@localhost:5532/ai`; `vector`
  extension enabled at first boot.

## 3. Dependencies & Configuration

Add the data-tier dependencies and a small **Runtime-level config** surface — the
first of the top-level shared siblings v0.2 §1 reserved space for.

- **New runtime dependencies** (`[project.dependencies]`):
  - `sentence-transformers` — the in-process embedder (pulls `torch`).
  - `torch` pinned to a **cu128** build via `[tool.uv.sources]` so uv resolves the
    Blackwell-capable wheel rather than the default CPU/older-CUDA one.
  - `sqlalchemy`, `psycopg[binary]`, `pgvector` — the Postgres + pgvector drivers
    Agno's `PgVector` requires.
  - `pydantic-settings` — typed env-driven configuration.
- **`config.py`** (top-level, alongside `app.py`): a `pydantic-settings`
  `Settings` reading env — `DATABASE_URL`, `EMBEDDER_MODEL` (default
  `Qwen/Qwen3-Embedding-8B`), `EMBEDDER_DEVICE` (`cuda` / `cpu`),
  `EMBEDDER_DIMENSIONS` (default `4096`). A committed `.env.example` documents the
  keys; real `.env` is git-ignored.
- **CI vs. dev divergence is config, not code.** CI points `EMBEDDER_MODEL` at a
  tiny CPU model (or skips embedder tests entirely, §10); dev uses the 8B model on
  `cuda`. The wiring code is identical — only env differs.

**Deliverables:**

- [ ] Deps added and `uv.lock` updated (torch resolves to the cu128 build);
  `config.Settings` + `.env.example` present.

## 4. Database & Knowledge Wiring

Promote the deferred shared components into top-level siblings of `recipe_agent/`
(per v0.2 §1) and wire them to real infrastructure.

- **`db.py`** — constructs a `PostgresDb` from `Settings.DATABASE_URL`. In v0.3 it

- **`knowledge.py`** — constructs `Knowledge(vector_db=PgVector(...))` with:

  - `search_type=SearchType.vector` — **vector-only** for now (Chinese keyword

  agents reuse the same knowledge/db handles.

**Deliverables:**

## 5. HowToCook Vendoring

Phase A source (design §2.2): `Anduin2017/HowToCook`, public-domain templated
Markdown. Bring it in as a **pinned, offline snapshot** — no network at ingest
time.

- Vendor the repo's `dishes/` tree at a **pinned commit** (recorded for
  provenance/reproducibility). A re-fetch is an explicit, reviewed bump, never an
  implicit moving target.
- Only the recipe Markdown under `dishes/` is needed; build tooling and unrelated
  docs are excluded.

**Deliverables:**

- [ ] HowToCook `dishes/` vendored at a recorded pinned commit; ingest reads from
  the local snapshot with zero network.

## 6. HowToCook Markdown → `Recipe` Parser

Implement `HowToCookIngester.ingest()` (the v0.2 stub): walk the vendored tree and
normalize each templated Markdown file into a typed `Recipe` (design §3).

- **Stream, don't list** — `ingest()` yields `Recipe` objects lazily over the
  tree (the v0.2 `Iterator[Recipe]` contract), so a large source never sits fully
  in memory.
- **Map the template to the model:**
  - the ingredient table/list → **structured `ingredients`** (`name`, raw-string
    `quantity` preserving ranges like `10-15`, `unit`, `role` defaulting to
    `main`, `optional`).
  - the steps section → `steps` as **Markdown**, structure preserved (design §3).
  - `name`, `category` (from the `dishes/<category>/` path), `servings` /
    `difficulty` where the template states them; `provenance =
    RecipeProvenance(source="howtocook", ref=<relative path>)`.
- **Images** are recorded as runtime-resolvable refs (design §3); actually copying
  / serving media is out of scope — refs may point at the vendored path.
- Parsing is **best-effort and lossless-leaning**: unparseable scalars stay
  `None`; raw Markdown is never discarded.

**Deliverables:**

- [ ] `HowToCookIngester.ingest()` yields valid `Recipe` objects for the vendored
  snapshot; parser covered by fixture-based unit tests (§10).

## 7. `RecipeSink` Persistence

Implement `RecipeSink.write()` (the v0.2 stub): drain a `Recipe` stream into the
knowledge base, idempotently.

- **Recipe → knowledge content.** Each `Recipe` becomes one knowledge entry: a
  text body (name + description + ingredients + steps, rendered for embedding) plus
  **metadata** (`id`, `category`, `provenance`, difficulty, etc.) for later
  filtering. The embedder vectorizes the body; the vector + metadata upsert into
  pgvector.
- **Idempotency via the contents-db.** A stable content key derived from
  `Recipe.id` / provenance lets a re-run **skip** unchanged recipes rather than
  duplicate them — this is what makes `skipped` meaningful and re-ingestion safe.
- **Honest counts.** `WriteResult` reports `written` (new), `skipped` (already
  present, unchanged), `failed` (could not embed/persist) — no exceptions
  swallowed silently; failures are counted and logged.

**Deliverables:**

- [ ] `RecipeSink.write()` embeds + upserts into pgvector with correct
  `written/skipped/failed`; a second run on the same input writes 0 / skips all.

## 8. Ingestion Entry Point

Ingestion is a **one-shot batch job**, not a request-path concern — expose it as a
command, not an HTTP route.

- A CLI entry point (e.g. `python -m spectres_runtime.recipe_agent.ingestion` or a
  `[project.scripts]` console script) wires `HowToCookIngester` → `RecipeSink` and
  prints the resulting `WriteResult`.
- Re-running it is safe (idempotent, §7); it is the operational path to
  (re)populate the knowledge base.

**Deliverables:**

- [ ] One command populates the knowledge base end-to-end and reports counts.

## 9. Retrieval Verification

Prove the data is **retrievable** — the whole point of v0.3 — without any chat
model.

- A check (script or `integration` test) issues a Chinese-language query (e.g.
  「番茄炒蛋怎么做」) against the knowledge base via Agno's native search
  (`knowledge.search` / `search_knowledge_base`) and asserts a relevant recipe is
  returned in the top results.
- This exercises the full embed → store → embed-query → vector-search loop. It is
  **retrieval grounding, not generation** — no `Agent.run`, no live LLM.

**Deliverables:**

- [ ] A retrieval check returns a relevant recipe for a Chinese query against the
  populated knowledge base.

## 10. Testing & Quality

v0.3 introduces a database and a heavy model, so the single flat gate from
v0.1/v0.2 splits into **two tiers**. The default gate stays pure; everything
infrastructural is opt-in.

- **Unit tier (default CI, keeps the v0.1 bar).** Pure, fast, deterministic — no
  DB, network, GPU, or model download. Covers the parser (fixture Markdown →
  `Recipe`), the `Recipe → content` mapping, and `WriteResult` accounting (with a
  fake/in-memory sink). `ruff` / `mypy --strict` / `pytest`, coverage ≥ 80% on the
  unit-testable code.
- **Integration tier (`@pytest.mark.integration`, opt-in).** Exercises real
  pgvector (a disposable container) and a **small CPU embedder** — never the 8B
  GPU model — to keep it runnable without a 5090. Covers `RecipeSink` upsert +
  idempotency and the §9 retrieval check. Excluded from the default `pytest` run
  and from the coverage gate; run explicitly (`pytest -m integration`) locally and
  in an optional CI job.
- **Marker registered** in `pyproject.toml` (`[tool.pytest.ini_options].markers`)
  so `--strict-markers` accepts it; default `addopts` add `-m "not integration"`.
- Coverage stays honest: embedder/DB code paths that only run under integration
  are scoped out of the unit coverage denominator (per-file omit or `pragma`),
  rather than padding the unit gate.

**Deliverables:**

- [ ] Unit tier green and pure (no DB/network/GPU), coverage ≥ 80%; `integration`
  tier defined, marker registered, excluded from default CI, runnable locally.

## Explicitly Out of Scope for v0.3

v0.3 is the **data/retrieval** slice only. Everything conversational and personal
is deferred.

- **Live chat model + the agent's RAG composition** — wiring a real domestic
  provider model, `search_knowledge_base` as an agent tool, instructions, and the
  meal-planning user journeys (design §5–§6). The placeholder model stays; no
  `Agent.run` against an LLM. → v0.4.
- **All personalization / User Profile work** — Profile Management, `User` /
  `Household`, merged constraints, `dependencies` injection, tools, User Memory
  (design §4).
- **ANN vector index** — at HowToCook scale exact search suffices; an index is not
  built (and 4096-dim cannot be indexed anyway, see Resolved Questions).
- **Chinese full-text / hybrid search** — vector-only for v0.3; `zhparser` /
  `pg_jieba` FTS and `SearchType.hybrid` deferred.
- **Reranking**, **media/object store** (image serving), **app containerization /
  CD**, **Alembic migrations**, **Auth / JWT / RBAC**, **Edge / Client**.
- **Data sources beyond HowToCook** — LLM-intrinsic (Phase B) and the Chinese
  platforms (Phase C) stay deferred (design §2.2).

## Resolved Questions

- **Embedder: which model and how is it run?** `SentenceTransformerEmbedder` with
  **`Qwen/Qwen3-Embedding-8B`** (4096-dim), run **in-process** (not Ollama / a
  separate service). In-process wins here: the ingest is a one-shot offline batch
  with no online-latency need, it keeps one fewer moving part, and it gives full
  control over model choice, full precision, and the query/passage prefixing that
  Chinese embedding quality depends on. Quality is a function of the *model*, not
  the runtime; in-process makes the best models (and full fp precision) easiest to
  reach.
- **Why a Chinese/large model at all?** The source (HowToCook) is Chinese.
  English-default models (e.g. `bge-small-en`) retrieve poorly on it. The 5090
  comfortably runs an 8B-class model, so v0.3 does not compromise on retrieval
  quality.
- **4096 dimensions and pgvector indexing.** pgvector ANN indexes cap out at 2000
  dims (`ivfflat`/`hnsw` on `vector`) / 4000 dims (`halfvec`) — **4096 cannot be
  ANN-indexed at all**. This is fine: at HowToCook scale (~1–2k recipes) pgvector
  **exact** (un-indexed) KNN is millisecond-fast and *more* accurate than ANN. So
  v0.3 stores 4096-dim vectors and searches them exactly, no index. (If the corpus
  later grows past exact-search comfort, reduce dimensionality via the model's MRL
  support — e.g. request 2048 dims — to fit `halfvec` indexing; out of scope now.)
- **Search type: vector-only.** `SearchType.hybrid` adds a PostgreSQL `tsvector`
  keyword leg, but the default text-search config does not tokenize Chinese, so the
  keyword half would be near-useless without `zhparser`/`pg_jieba`. v0.3 uses
  **vector-only** search and defers Chinese FTS rather than ship a broken hybrid.
- **GPU is a dev convenience, not a Runtime dependency.** Ingestion is the only GPU
  consumer and it is an offline, occasional batch. The Runtime service itself needs
  no GPU; CI never does (it uses a small CPU embedder or skips the integration
  tier).

## Open Questions

- **Vendoring mechanism — git submodule vs. pinned fetch script.** A submodule
  records the exact commit natively but adds submodule workflow friction; a fetch
  script (download a pinned tarball into a git-ignored path) keeps the working
  tree clean but needs the pin recorded elsewhere. Decide during §5.
- **Embedding text shape.** Exactly which `Recipe` fields compose the embedded
  body (and in what order/format) materially affects retrieval; to be tuned with
  the §9 check against real queries.
