# v0.3 — Recipe Ingestion & Knowledge Base

**Status:** planning draft. Architecture and rationale live in the design doc
[`docs/design/recipe-agent.md`](../design/recipe-agent.md); this is the v0.3 work
breakdown. The prior skeleton is
[`v0.2-recipe-agent-skeleton.md`](./v0.2-recipe-agent-skeleton.md). Runtime-wide
concepts (AgentOS, Postgres + pgvector, Models) are defined in
[`Agents.md`](../../Agents.md).

**Goal:** Give the v0.2 skeleton its **first real behavior on the data side**:
turn the behavior-empty stubs into a working ingestion path that fills a
**searchable recipe knowledge base**. v0.3 stands up Postgres + pgvector, wires a
**hosted embedding API** (a Chinese-strong, OpenAI-compatible service — SiliconFlow
`Qwen/Qwen3-Embedding-0.6B` as the working default; final model confirmed at §9, see
Open Questions), implements the **HowToCook** ingester (vendored snapshot → Markdown → `Recipe`),
and persists each recipe through `RecipeSink` into pgvector — then proves the data
is retrievable via Agno's native knowledge search.

Where v0.2 was *"structurally-complete but behavior-empty"*, v0.3 is
**"data-complete but conversation-empty"**: real recipes are parsed, embedded,
stored, and retrievable — but the agent is **still not wired to a live chat
model** and answers no user journeys. Live LLM traffic, the agent's RAG
composition, and **all personalization** stay deferred to v0.4+ (see Out of
Scope). Isolating ingestion gives v0.3 a **deterministic** acceptance bar (N
recipes in, relevant recipe out) with no LLM-evaluation subjectivity.

**Definition of Done:**

- The embedding provider is configured and reachable (§1+§3): an `EMBEDDER_API_KEY`
  is wired through `Settings`, and the embedder returns a vector of the configured
  dimensionality for a Chinese test string.
- `docker compose up` brings up Postgres + pgvector; the app connects via a
  configured `DATABASE_URL` (§2–§3).
- `HowToCookIngester.ingest()` yields typed `Recipe` objects parsed from a
  **vendored, pinned** HowToCook snapshot; `RecipeSink.write()` embeds and upserts
  them into pgvector, returning honest `written / skipped / failed` counts and
  **idempotent** on re-run (§5–§7).
- A single ingestion command populates the knowledge base; a retrieval check
  returns a relevant recipe for a Chinese-language query — **no chat LLM
  involved** (§8–§9).
- Gates stay green with a **new test tier split** (§10): pure unit tests (parser,
  mapping) keep the v0.1 bar (no DB / network, coverage ≥ 80%); DB- and
  embedder-dependent tests are marked `integration` and run opt-in.

## 1. Embedding Provider (hosted API)

Embeddings come from a **hosted, OpenAI-compatible API**. This is a deployment
decision: a hosted embedding API scales on demand, needs no GPU or model-weight
ops, and is cheap per call. Crucially, both **passages (ingest) and queries
(search) must use the same embedder** (same model = same vector space), so this one
choice governs the whole retrieval loop.

- **Chat model ≠ embedder.** The v0.4 chat model (Kimi / Moonshot) has **no
  embeddings endpoint** — it only generates text. RAG still needs a dedicated
  embedder for the query vector, sourced separately from the chat LLM.
- **Working default: SiliconFlow `Qwen/Qwen3-Embedding-0.6B`** (Alibaba Qwen3
  embedding family, Apache-2.0 open weights, hosted on SiliconFlow / 硅基流动).
  Chinese-strong (CMTEB-R ≈ 71, well ahead of `BAAI/bge-m3` at the same ~0.6B size
  and 1024 dims), OpenAI-compatible, instruction-aware, and — unlike `bge-m3` on
  SiliconFlow — it **honors the `dimensions` request param** (MRL, 32–1024). The
  wiring is provider-agnostic via Agno's `OpenAIEmbedder` + `base_url`, so swapping to
  a higher-quality sibling (4B/8B) or another vendor (DashScope `text-embedding-v4`,
  itself the commercial wrapper of this same Qwen3 family) is a **config change**, not
  a code change. Final model is confirmed against real Chinese queries at §9.
- **Mainland base URL.** Use `https://api.siliconflow.cn/v1` (the `.cn` host; the
  international `.com` host is a separate account/key namespace).
- **The model name is a data contract.** Stored vectors are only queryable by the
  *same* model that produced them. Record the embedder model in provenance and
  treat a provider model change/deprecation as a **full re-embed** event.
- **No GPU anywhere.** Embeddings are a network call; neither ingestion, the
  Runtime, nor CI needs a local GPU or model weights. (The model being open-weight
  keeps a future self-host path open with the *same* vector space — no re-embed.)

**Deliverables:**

- [x] `EMBEDDER_API_KEY` wired through `Settings`; the configured embedder returns
  a vector of the expected dimensionality for a Chinese test string.

## 2. Infrastructure: Postgres + pgvector

Stand up the single datastore `Agents.md` mandates — Postgres + pgvector, the
source of truth for knowledge vectors (and, later, sessions / memory / profile).

- **`docker/compose.yaml`** runs Agno's official **`agnohq/pgvector:16`** image
  (Postgres 16 with the `vector` extension preinstalled), with a named volume for
  durability and a `pg_isready` health check. This is the only service v0.3 needs.
  Follows Agno's host-port convention (**5532**→5432) so it never clashes with a
  standard local Postgres or sibling projects on 5432.
- **App containerization is *not* in scope.** The Runtime still runs via `uvicorn`
  on the host; only the database is containerized. A full app image / compose
  topology is deferred.
- The `vector` extension is enabled on first boot via a committed init SQL
  (`docker/postgres/init.sql`, mounted into `/docker-entrypoint-initdb.d`) so the
  DB is self-sufficient — independent of Agno's table-creation path, which also
  ensures it.

**Deliverables:**

- [x] `docker compose -f docker/compose.yaml up -d` yields a healthy Postgres +
  pgvector reachable at `postgresql+psycopg://developer:<password>@localhost:5532/spectres_runtime`; `vector`
  extension enabled at first boot.

## 4. Database & Knowledge Wiring

Promote the deferred shared components into a **`storage/`** package sitting
**level with `recipe_agent/`** (per v0.2 §1) — the Runtime's data/persistence layer
(name borrowed from `Agents.md`'s `## Storage` section) — and wire them to real
infrastructure. `storage/` holds DB-related wiring only (`db.py`, `knowledge.py`);
domain logic such as `profile/` stays out, even though it later *uses* the db.

- **`storage/db.py`** — constructs the Runtime's single shared `PostgresDb` from
  `Settings.DATABASE_URL`. This one handle is **Runtime-wide, not recipe-private**:
  per `Agents.md` it is the single source of truth for *all* persistent state
  (sessions, memory, profile, knowledge-content tracking, traces), each occupying
  its own table in one database. "Contents DB" is not a separate database but a
  **role** — the slot a `PostgresDb` plugs into on `Knowledge` (`contents_db=`,
  table `agno_knowledge`). In v0.3 this shared handle activates **only** that
  one role: tracking what was ingested, which is what makes §7's idempotent `skipped`
  possible. The sessions / memory / profile roles are the *same* handle gaining
  *more tables* later — deferred to the agent release, not a different db.
- **`storage/knowledge.py`** — constructs `Knowledge(vector_db=PgVector(...))` with:
  - `table_name="recipes"`, `db_url` from settings.
  - `embedder` from `Settings.build_embedder()` (the §1/§3 `OpenAIEmbedder` —
    SiliconFlow Qwen3-Embedding — **reused**, so ingest and search share one vector
    space rather than constructing a second embedder).
  - `search_type=SearchType.vector` — **vector-only** for now (Chinese keyword
    FTS deferred, see Resolved Questions).
  - `contents_db` set to the shared `PostgresDb` from `storage/db.py` (the §7
    idempotency role).
- These are **Runtime-level shared components**, not recipe-private — future
  agents reuse the same knowledge/db handles.

**Deliverables:**

- [x] `storage/db.py` + `storage/knowledge.py` construct against a live Postgres +
  pgvector; `recipes` table + `vector` column created with the correct dimensionality.

## 6. HowToCook → `Recipe`: offline enrichment + ingest

The `必备原料和工具` section is **not regular** — it mixes tools (`擀面杖`,
`平底锅`) with ingredients, carries free-text notes and sub-headers (`可选（巧克力
味）`), lumps several items on one line (`生抽、蚝油、盐`), and uses inconsistent
bullets — so a deterministic parser handles the bulk but leaves a residue of
semantic edge cases (tool-vs-ingredient, brand/over-description trimming, `A或B`
alternatives) that need judgment. The work is therefore split into an **offline
enrichment build** — a deterministic extraction script plus a one-time, reviewed
curation pass — that emits a committed structured catalog, and a **deterministic,
zero-network ingest** that reads it.

### 6.1 Offline enrichment → `datasets/howtocook/ai-cleaned/recipes.jsonl`

A committed extraction script processes the snapshot into one catalog,
written under a dedicated `datasets/howtocook/ai-cleaned/` directory (its name
marking it as the cleaned derivative, kept separate from the faithful
`dishes/` snapshot) — a **JSONL** file, **one recipe per line**, entries
**sorted by `ref`**. JSONL (over a single JSON array) is chosen because the
catalog is a committed artifact reviewed on every tag bump: a line-per-recipe
layout keeps git diffs legible (one changed recipe = one changed line) and a
stable `ref` sort makes regeneration deterministic, while still loading whole in
one pass. Per line:

- `ref` — relative path under `dishes/` (the join key back to the snapshot; recorded
  as `provenance.ref` and the basis of the recipe's stable `id`).
- `name` — the `.md` filename stem of `ref` (content-independent; not the H1).
- `images` — list of relative image refs.
- `difficulty` — `★` count (1–5).
- `description` — the intro paragraph.
- `ingredients` — list of `{name, optional}`; **tools removed**, free text dropped,
  one-item-per-line split, deduped.

All five scalar/list fields **and** `ingredients` are produced by a single
**deterministic** extraction script, `scripts/howtocook_clean.py` (tools dropped
via a curated keyword/whitelist list, quantities/notes stripped, multi-item
bullets split on `、，,/`, optional flagged). A subsequent **one-time,
hand-reviewed curation pass** then refines only the ingredient names where
deterministic rules fall short — trimming brand/over-description to the bare item
(`闽星茶树菇`→`茶树菇`, `叶菜类蔬菜`→`蔬菜`), splitting `A或B` / `or` / `二选一`
alternatives (the second onward marked optional), and dropping residual tools.
We **do not** parse quantity/unit: amounts stay in the recipe body (`## 计算`)
for the cook to read and balance.

- **Committed derived artifact.** Like the snapshot, the catalog is committed and
  kept separate from the faithful `dishes/` snapshot. The deterministic pass is
  reproducible from `scripts/howtocook_clean.py`; the subsequent manual curation
  makes the final `recipes.jsonl` **no longer byte-reproducible from the script by
  design** — regenerating then re-curating is an explicit, reviewed step on a tag
  bump.
- **Source-grounded names.** The deterministic pass takes ingredient names verbatim
  from the dish text; the one-time curation may rewrite a name to a cleaner form
  (brand-trim, split, simplify), so names are **review-grounded** rather than
  guaranteed verbatim substrings of the source.

### 6.2 Ingest: `HowToCookIngester.ingest()`

Implement the v0.2 stub to yield typed `Recipe` objects (design §3) from the local
artifacts — no parsing of the irregular ingredient section at runtime.

- **Stream, don't list** — `ingest()` yields `Recipe` lazily (the v0.2
  `Iterator[Recipe]` contract). The catalog is small enough to load whole; the bulk
  step Markdown is read per recipe.
- **Compose each `Recipe`:** structured fields (`name`, `images`, `difficulty`,
  `ingredients` as `{name, optional}`) from the catalog entry; `category` from the
  `dishes/<category>/` path; `steps` as **Markdown** read from the snapshot `.md`
  (the `## 计算` quantity table onward, falling back to `## 操作`), structure
  preserved (design §3); `provenance =
  RecipeProvenance(source="howtocook", ref=<ref>)`; `id =
  f"{source}/{ref without .md}"` (source-namespaced, slug-style, e.g.
  `howtocook/aquatic/咖喱炒蟹`, so ids stay unique across origins; the raw `ref`
  lives in `provenance.ref`).
- Best-effort and lossless-leaning: missing scalars stay `None`; raw step Markdown
  is never discarded.

> **Model note:** quantity/unit are dropped — `Ingredient` is `{name,
> optional}` (`models/ingredient.py`). Amounts stay in the recipe body.

**Deliverables:**

- [x] `scripts/howtocook_clean.py` + a one-time hand-curation produce
  `datasets/howtocook/ai-cleaned/recipes.jsonl` (357 entries — all `dishes/`
  recipes except the `template/` placeholder) with clean `{name, optional}`
  ingredients.
- [x] `HowToCookIngester.ingest()` yields valid `Recipe` objects from the catalog +
  snapshot; covered by fixture-based unit tests (§10).

## 7. `RecipeSink` Persistence

Implement `RecipeSink.write()` (the v0.2 stub): drain a `Recipe` stream into the
knowledge base, idempotently.

- **Recipe → knowledge content.** Each `Recipe` becomes one knowledge entry: a
  text body (name + description + ingredients + steps, rendered for embedding) plus
  **metadata** (JSONB stored alongside the vector). Only the body is embedded — the
  body alone governs retrieval quality (the §10 "embedding text shape" question).
  Metadata does **not** improve recall; it serves three other roles: (1) a stable
  **content key** for idempotency (§7 below), (2) **provenance** — `source`, `ref`,
  and the **embedder model name** (the re-embed data contract, §1), and (3) **filter
  + display payload** — `category`, `difficulty`, image refs — stored now
  (near-zero cost) so v0.4 can do metadata-filtered search ("素菜只", "30分钟内") and
  render results without re-walking the corpus. Structured-only fields (`id`,
  `difficulty`, image paths) go to metadata, not the embedded body, to avoid diluting
  the vector. The embedder vectorizes the body; the vector + metadata upsert into
  pgvector.
- **Idempotency via the shared db's contents tracking.** The `agno_knowledge`
  table (the shared `PostgresDb` in its `contents_db` role, §4) records what was
  ingested; a stable content key derived from `Recipe.id` / provenance lets a re-run
  **skip** unchanged recipes rather than duplicate them — this is what makes
  `skipped` meaningful and re-ingestion safe.
- **Honest counts.** `WriteResult` reports `written` (new), `skipped` (already
  present, unchanged), `failed` (could not embed/persist) — no exceptions
  swallowed silently; failures are counted and logged.

**Deliverables:**

- [ ] `RecipeSink.write()` embeds + upserts into pgvector with correct
  `written/skipped/failed`; a second run on the same input writes 0 / skips all.

## 10. Testing & Quality

v0.3 introduces a database and an external embedding API, so the single flat gate
from v0.1/v0.2 splits into **two tiers**. The default gate stays pure; everything
infrastructural is opt-in.

- **Unit tier (default CI, keeps the v0.1 bar).** Pure, fast, deterministic — no
  DB, no network, no API key. Covers the parser (fixture Markdown →
  `Recipe`), the `Recipe → content` mapping, and `WriteResult` accounting (with a
  fake/in-memory sink and a **mocked embedder**). `ruff` / `mypy --strict` /
  `pytest`, coverage ≥ 80% on the unit-testable code.
- **Integration tier (`@pytest.mark.integration`, opt-in).** Exercises real
  pgvector (a disposable container) and the **real embedding API** (gated on
  `EMBEDDER_API_KEY`; skipped when unset). Covers `RecipeSink` upsert +
  idempotency and the §9 retrieval check. Excluded from the default `pytest` run
  and from the coverage gate; run explicitly (`pytest -m integration`) locally and
  in an optional secret-gated CI job.
- **Marker registered** in `pyproject.toml` (`[tool.pytest.ini_options].markers`)
  so `--strict-markers` accepts it; default `addopts` add `-m "not integration"`.
- Coverage stays honest: embedder/DB code paths that only run under integration
  are scoped out of the unit coverage denominator (per-file omit or `pragma`),
  rather than padding the unit gate.

**Deliverables:**

- [ ] Unit tier green and pure (no DB/network), coverage ≥ 80%; `integration`
  tier defined, marker registered, excluded from default CI, runnable locally.

## Explicitly Out of Scope for v0.3

v0.3 is the **data/retrieval** slice only. Everything conversational and personal
is deferred.

- **Live chat model + the agent's RAG composition** — wiring a real domestic
  provider model, `search_knowledge_base` as an agent tool, instructions, and the
  meal-planning user journeys (design §5–§6). The placeholder model stays; no
  `Agent.run` against an LLM. → v0.4.
- **All personalization / User Profile work** — Profile Management, `User` /
  `Household`, merged constraints, `dependencies` injection, tools, User Memory
  (design §4).
- **ANN vector index** — at HowToCook scale (~1–2k recipes) exact (un-indexed) KNN
  is millisecond-fast and more accurate, so no index is built in v0.3. (The default
  1024-dim vector is `ivfflat`/`hnsw`-indexable later if the corpus grows — see
  Resolved Questions. Note this is *why* the default stays ≤2000: a 4B/8B sibling at
  2560/4096 dims would exceed pgvector's 2000-dim index ceiling for the `vector`
  type.)
- **Chinese full-text / hybrid search** — vector-only for v0.3; `zhparser` /
  `pg_jieba` FTS and `SearchType.hybrid` deferred.
- **Reranking** — a cross-encoder second stage (e.g. `Qwen/Qwen3-Reranker-0.6B`,
  same family + platform) is a **precision** optimization on top of a working
  first-stage retrieval, not a baseline component. It is deferred deliberately: add
  it only if the §9 check (or real v0.4 usage) shows the right recipe consistently
  lands outside the top-3. Crucially it is **non-destructive** — a reranker re-scores
  candidates *after* search and needs **no re-embed**, so deferring it costs nothing.
- **Media/object store** (image serving), **app containerization /
  CD**, **Alembic migrations**, **Auth / JWT / RBAC**, **Edge / Client**.
- **Data sources beyond HowToCook** — LLM-intrinsic (Phase B) and the Chinese
  platforms (Phase C) stay deferred (design §2.2).

## Resolved Questions

- **Embedder: hosted API.** A **hosted, OpenAI-compatible embedding API**
  (`OpenAIEmbedder` + `base_url`) is used for both ingest and serving. Query
  embedding happens on every search, and a hosted API scales on demand, needs no
  GPU or model-weight ops, and costs little per call. The recipe corpus is public
  (HowToCook), so sending it to the API raises no data-sensitivity concern.
- **Same embedder for passages and queries — non-negotiable.** Vector search
  compares distances *within one model's vector space*. Embedding passages with one
  model and queries with another yields incompatible coordinates and broken
  retrieval, so a single embedder governs the whole loop. A provider model
  change/deprecation invalidates stored vectors and forces a full re-embed — hence
  the model name is pinned in provenance.
- **Why this Chinese-strong model.** The source (HowToCook) is Chinese;
  English-default models retrieve poorly on it. SiliconFlow `Qwen/Qwen3-Embedding-0.6B`
  is from the Qwen3 embedding family (June 2025, Apache-2.0), which tops CMTEB in its
  size class: at the same ~0.6B size and 1024 dims it scores CMTEB-R ≈ 71 vs `bge-m3`
  (a 2024 model) markedly lower, while also being the only one of the two whose
  SiliconFlow endpoint honors the `dimensions` param. It is the open-weight twin of
  DashScope `text-embedding-v4`, unifying the earlier "open-weight vs. commercial"
  candidates into one. (Final confirmation is the §9 real-query check — see Open
  Questions.)
- **Dimensions and pgvector indexing.** Qwen3-Embedding-0.6B emits configurable
  dimensions (MRL, 32–1024); v0.3 defaults to **1024**. Because that is ≤2000, it
  *can* be `ivfflat`/`hnsw`-indexed later if the corpus grows. v0.3 still uses
  **exact** (un-indexed) KNN: at HowToCook scale it is millisecond-fast and more
  accurate than ANN. A larger sibling (4B@2560 / 8B@4096) would raise quality but
  break the ≤2000 index ceiling for the `vector` type — hence 0.6B@1024 is the
  default sweet spot; a 4B model truncated via MRL to ≤1536 is the fallback if §9
  shows 0.6B is insufficient.
- **Metadata vs. embedding text.** Only the rendered text body is embedded and
  governs retrieval; metadata (JSONB beside the vector) is for idempotency,
  provenance (incl. the pinned model name), filtering, and display — it does not
  affect recall (§7).
- **Reranking deferred, not rejected.** First-stage vector retrieval with a
  top-class embedder is expected to suffice at this corpus size; a reranker is a
  non-destructive, re-embed-free add-on to bolt on later only if §9/usage data shows
  precision@3 is weak (see Out of Scope).
- **Search type: vector-only.** `SearchType.hybrid` adds a PostgreSQL `tsvector`
  keyword leg, but the default text-search config does not tokenize Chinese, so the
  keyword half would be near-useless without `zhparser`/`pg_jieba`. v0.3 uses
  **vector-only** search and defers Chinese FTS rather than ship a broken hybrid.
- **Vendoring mechanism: committed snapshot.** The `dishes/` Markdown is committed
  directly into `datasets/howtocook/dishes/` (not a submodule, not a runtime fetch).
  A plain `git clone` is a true zero-network snapshot with no extra workflow.
  Upstream's The Unlicense (public domain) makes direct inclusion unrestricted. The
  pin (tag `1.6.0`) is recorded in §5 and reproduced by `scripts/howtocook_download.py`.

## Open Questions

- **Final embedding model — confirm 0.6B at §9.** `Qwen/Qwen3-Embedding-0.6B` @ 1024
  is the working default and the expected final choice; the open item is the **§9
  real-query check** (Chinese, conversational, against the actual recipe corpus —
  not just public CMTEB scores). If precision is weak, the provider-agnostic wiring
  makes the escalation a config change: `Qwen3-Embedding-4B` (MRL-truncated to ≤1536
  to stay index-able) or, separately, adding a reranker. Settle before bulk ingest,
  since changing the embedder means re-embedding.
- **Embedding dimensionality.** Default `1024` (0.6B's native max). Whether a larger
  model at a higher dim repays the cost/latency/index tradeoff is decided together
  with the model question at the §9 check.
- **Embedding text shape.** Exactly which `Recipe` fields compose the embedded
  body (and in what order/format) materially affects retrieval; to be tuned with
  the §9 check against real queries.
