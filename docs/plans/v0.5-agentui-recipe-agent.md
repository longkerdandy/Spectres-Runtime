# v0.5 — AgentUI + Recipe Agent (Barely Usable Demo)

**Status:** planning draft. Architecture and rationale live in the design doc
[`docs/design/recipe-agent.md`](../design/recipe-agent.md); this is the v0.5 work
breakdown. The prior slice that made the agent talk is
[`v0.4-recipe-agent-live.md`](./v0.4-recipe-agent-live.md). Runtime-wide concepts
(AgentOS, Models, Postgres + pgvector) are defined in [`Agents.md`](../../Agents.md).

**Goal:** Make the v0.4 recipe agent **demonstrable in a real web UI**.
v0.5 layers Agno's official **[AgentUI](https://docs.agno.com/other/agent-ui)**
on top of the live recipe agent, so a user can open a browser, connect to the
local AgentOS, and hold a grounded, multi-turn Chinese recipe conversation.
This is the first version where the recipe agent is **"barely usable"**:
it looks like a product, not a curl script.

Where v0.4 was *"conversation-complete but UI-empty"*, v0.5 is
**"UI-complete but personalization-empty"**: the agent answers recipe
questions in a chat interface, keeps multi-turn context, and formats replies
as readable Markdown — but it still has **no saved meal plans**, **no household
profile**, and **no user memory**. Those remain deferred (see Out of Scope).

**Definition of Done:**

- Agno's official `AgentUI` frontend is wired into the repository and connects
  to the local AgentOS on first open (§1–§2).
- A single dev command launches the database, the backend, and the frontend
  together (§2).
- The backend exposes the AgentOS surface with CORS configured for the dev UI
  origin (§3).
- The recipe agent emits **formatted Markdown** (headings, lists, tables) that
  renders cleanly in AgentUI (§4).
- The full path is verified end-to-end through AgentUI: a user asks
  「番茄炒蛋怎么做」, gets a grounded Markdown answer, then asks
  「那它要炒几分钟？」 and the follow-up resolves using session history
  (§5).
- Telemetry stays off end to end; no anonymous run data leaves the process.
- Gates stay green on the **two-tier split** from v0.3 §10 / v0.4 §7 (§6):
  unit tests assert agent shape and CORS wiring with no live model call
  (coverage ≥ 80%); the live UI walkthrough is a manual/integration check,
  excluded from default CI.

## 1. AgentUI Frontend Setup

Pull in Agno's official open-source chat UI as the dev/demo frontend. The
Runtime remains backend-only in production, but shipping a first-class local
UI makes the agent instantly explorable and surfaces formatting issues early.

- **Location:** `agent-ui/` at the repo root. Created with Agno's official
  scaffold:
  ```bash
  npx create-agent-ui@latest agent-ui
  ```
  The generated app is committed **except** build artifacts and dependencies
  (`node_modules/`, `.next/`, `dist/`, `.turbo/`, lockfiles if they churn).
- **Git hygiene:** Add the standard Node.js excludes to the root `.gitignore`
  so the frontend can live in-repo without polluting diffs. Do **not** treat
  `agent-ui/` as a submodule — a plain `git clone` must yield a runnable UI
  after `npm install`.
- **Default endpoint.** Patch the generated scaffold so the UI's default
  AgentOS endpoint points to the Runtime's dev port (`http://localhost:8000`).
  The user can still override it in the sidebar, but the first-run experience
  should be "open localhost:3000 and it just works".
- **No custom React components in v0.5.** Use AgentUI out of the box. Custom
  widgets (recipe cards, image carousels) are deferred to a later frontend
  slice; v0.5 only needs the stock chat surface.

**Deliverables:**

- [ ] `agent-ui/` directory committed, scaffolded from `create-agent-ui@latest`,
  with `node_modules/` and build artifacts ignored.
- [ ] Default AgentOS endpoint in the UI points to `http://localhost:8000`
  (or the port the Runtime dev server binds).

## 2. One-Command Local Dev Launch

A demo user should not need four terminal windows. Provide a single script that
brings up Postgres, the Runtime backend, and the AgentUI frontend.

- **`scripts/dev.sh`** (Unix; no PowerShell required in v0.5) does the following
  in a predictable order:
  1. `docker compose --env-file .env -f docker/compose.yaml up -d`
     (ensure Postgres is healthy via `pg_isready` poll).
  2. `uv run recipe-ingest` if the recipes table is empty (detect via a lightweight
     SQL probe; this step is **best-effort** and skipped if `EMBEDDER_API_KEY`
     is missing, so the script does not fail on a partial env).
  3. Start `uv run uvicorn spectres_runtime.app:app_factory --factory --reload --port 8000`
     in the background.
  4. `cd agent-ui && npm install && npm run dev` in the foreground.
  5. On `Ctrl-C`, shut down the background uvicorn and the Postgres container
     (or leave Postgres running — document the choice).
- The script is **idempotent**: re-running it picks up the existing Postgres
  volume and existing `node_modules`.
- Document the script in `docs/environment.md` as the recommended day-one flow.

**Deliverables:**

- [ ] `scripts/dev.sh` launches database → (optional ingest) → backend → frontend
  with one command.
- [ ] `docs/environment.md` updated with the v0.5 quick-start path.

## 3. Backend: CORS & Dev-Facing Surface

AgentUI runs on `localhost:3000` and calls the AgentOS HTTP API from the
browser. The backend must allow that origin.

- **CORS middleware.** Mount `CORSMiddleware` on the FastAPI app for the dev
  origin only. Keep it **config-driven** (env var `AGENT_UI_ORIGIN`, default
  `http://localhost:3000`) so the same binary can be pointed at a deployed UI
  later without a code change. Do **not** use `allow_origins=["*"]` outside of
  a deliberate local-dev override.
- **Port alignment.** `app.py:main()` already binds `127.0.0.1:8000`; AgentUI's
  default endpoint should match. Document that `7777` referenced in some Agno
  docs is only an example.
- **No auth in v0.5.** `authorization` stays off, as in v0.4. The dev server
  remains loopback-only; CORS is the only new exposure.
- **Health / discovery endpoints remain unchanged.** `GET /health`, `GET /agents`,
  `POST /agents/recipe/runs`, and the AgentOS sessions/knowledge surface are
  provided for free by AgentOS; we write no custom endpoints.

**Deliverables:**

- [ ] `AGENT_UI_ORIGIN` wired through `Settings` + `.env.example`; CORS mounted
  on the FastAPI app in `build_app()` (production-origins list supports a
  single configured origin).
- [ ] Unit test confirms preflight `OPTIONS` from the configured origin succeeds
  and an unconfigured origin is rejected.

## 4. Agent Output Formatting (Markdown)

The v0.4 agent answers correctly but may return dense plain text. In a chat UI,
formatting is usability. Enable Markdown output and tune instructions so replies
render as clean cards in AgentUI.

- **`markdown=True` on the Agent.** Add this to `build_recipe_agent()`. Agno
  then emits content as Markdown and AgentUI renders it accordingly.
- **Instruction update.** Extend the default `RECIPE_AGENT_INSTRUCTIONS` to ask
  the model for a consistent, UI-friendly shape when it answers from a recipe:
  - Dish name as a heading (`## 菜名`).
  - Ingredients as a bulleted list or small table.
  - Steps as a numbered list.
  - Optional notes (difficulty, time) in a short preamble.
  - Citations remain: name the dish/dishes the answer is grounded in.
- **No structured-output schema in v0.5.** Markdown is enough for "barely
  usable"; a typed `RecipeCard` response with custom frontend rendering is
  deferred.
- **Image references.** The knowledge metadata already carries `images`. The
  agent is instructed to **mention** images if relevant (e.g. "这道菜有配图，
  可在原料列表上方查看") but not to fabricate URLs. Actually serving images
  from the vendored snapshot is deferred (no static file host yet).

**Deliverables:**

- [ ] `build_recipe_agent()` sets `markdown=True`.
- [ ] Default instructions updated to request Markdown headings, ingredient
  list/table, numbered steps, and dish citation.
- [ ] Unit test asserts `markdown=True` and the expected instruction substring
  are present on the built agent.

## 5. End-to-End UI Verification

Prove the loop by actually using AgentUI, not by curl.

- **Prerequisites:** `.env` filled; database up; corpus ingested (`recipe-ingest`
  run at least once); `./scripts/dev.sh` started.
- **Steps:**
  1. Open `http://localhost:3000`.
  2. Confirm the endpoint sidebar shows `http://localhost:8000`; select the
     Recipe Agent.
  3. Send: 「番茄炒蛋怎么做」.
  4. Expect: a grounded Markdown answer with a heading for 西红柿炒鸡蛋,
     ingredient bullets, numbered steps, and a citation naming the dish.
  5. Send a follow-up in the same thread: 「那它要炒几分钟？」.
  6. Expect: the answer uses the prior turn (no need to restate "番茄炒蛋")
     and references the same dish.
- **Session persistence check:** Refresh the browser, re-open the same
  `session_id` (via AgentUI's session picker or by supplying it), and confirm
  the prior turns are still there — proving the Postgres-backed session store
  works across reconnects.
- This verification is **manual** for v0.5. Capture a short note in the plan
  (or a checklist item) confirming the walkthrough was performed; no checked-in
  browser test is required.

**Deliverables:**

- [ ] Manual walkthrough checklist completed: single-turn grounded answer,
  follow-up with context, session survives refresh.

## 6. Testing & Quality

Keep the two-tier split. The UI itself is not unit-tested (it is Agno's code),
but the wiring that enables it is.

- **Unit tier (default CI, pure).** No DB, no network, no live model.
  - Agent shape: `markdown=True`, instructions contain Markdown cues,
    `telemetry=False`, injected `knowledge`/`db` present.
  - CORS: `TestClient` `OPTIONS` preflight from `AGENT_UI_ORIGIN` succeeds;
    preflight from a different origin fails.
  - `build_app()` accepts CORS origin as an injected parameter or reads it from
    settings; tests use a test double rather than env.
  - Coverage ≥ 80% on unit-testable code.
- **Integration tier (`@pytest.mark.integration`, opt-in).**
  - Optional: a headless HTTP check that `POST /agents/recipe/runs` returns a
    streaming response that contains Markdown formatting markers (`##`, `1.`).
  - Otherwise the §5 manual walkthrough satisfies the integration bar for v0.5.
- **Lint/type gate.** `ruff` / `mypy --strict` remain green. The `agent-ui/`
  directory is excluded from Python tooling via existing `extend-exclude` or
  a new ruff/mypy ignore entry.

**Deliverables:**

- [ ] Unit tests cover `markdown=True`, instruction content, and CORS preflight
  behavior; coverage ≥ 80%.
- [ ] `agent-ui/` is ignored by `ruff` and `mypy`.
- [ ] Manual or integration verification confirms Markdown-grounded answers
  through AgentUI.

## Explicitly Out of Scope for v0.5

v0.5 is the **"first usable demo in a web UI"** slice only.

- **Custom frontend widgets / React components.** No recipe-card component,
  no image carousel, no step-by-step checklist widget. AgentUI is used stock.
- **Image serving.** The vendored `datasets/howtocook/dishes/` images are not
  exposed over HTTP; the agent may mention them in text but does not render them.
- **Stateful meal-plan journeys** (design §6) — saved, editable plan object and
  conversational plan adjustment. → deferred beyond v0.5.
- **User Memory / preference learning** ("likes spicy", "found it too sweet").
  → deferred beyond v0.5.
- **All Profile Management work** (`User` / `Household`, merged constraints,
  `dependencies` injection, dietary tools). This is a separate module, deferred
  per project requirement.
- **Auth / JWT / RBAC** (`authorization=True`), **os.agno.com control-plane**
  linkage, **Teams / Workflows**, **AgentFactory** / directory-driven
  construction, **Alembic migrations**, **containerization / CD**.

## Resolved Questions

- **Frontend choice: Agno AgentUI.** It is the official, open-source,
  self-hosted chat interface for AgentOS. One `npx create-agent-ui@latest`
  scaffold gives a working Next.js app; no custom React work is required for
  v0.5.
- **Backend port: 8000.** Aligns with the existing `app.py:main()` dev server.
  AgentUI's default endpoint is patched to match; documentation clarifies that
  `7777` in upstream examples is only illustrative.
- **CORS: config-driven, single origin.** `AGENT_UI_ORIGIN` defaults to
  `http://localhost:3000`. This keeps the dev server usable while avoiding a
  wildcard in any deployed configuration.
- **Formatting: Markdown only.** A typed `RecipeCard` schema + custom widget
  would look better but is overkill for "barely usable". `markdown=True` plus
  instruction-tuned headings/lists is the right v0.5 trade-off.
- **Images: text references only.** Serving images requires a static-file route
  and potential path mapping; deferred. The knowledge metadata already carries
  `images` for when the frontend slice arrives.

## Open Questions

- **Should `scripts/dev.sh` leave Postgres running after `Ctrl-C`?** Starting
  cold is slow because of `pg_isready` polling; leaving it up improves iteration
  time but may surprise users who expect a clean shutdown. Decide after using
  the script for a day and document the chosen behavior.
- **AgentUI lockfile strategy.** `create-agent-ui` emits a `pnpm-lock.yaml` by
  default. Decide whether to commit it (reproducible installs) or ignore it
  (smaller diffs, accept drift) — leaning toward committing for reproducibility.
- **Default instructions for follow-up formatting.** When the user asks a
  follow-up (e.g. 「那它要炒几分钟？」), the model may repeat the full recipe
  or answer tersely. Tune the instruction wording during §5 so the UI feels
  coherent without being verbose.
