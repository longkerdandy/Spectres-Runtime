# v0.5 — Runtime: Enable Spectres-Web (AgentUI) + Recipe Agent

**Status:** planning draft. Architecture and rationale live in the design doc
[`docs/design/recipe-agent.md`](../design/recipe-agent.md); this is the v0.5 work
breakdown for the **Runtime** side. The companion frontend plan lives in the
sibling repository `Spectres-Web` at `docs/plans/v0.5-agentui-recipe-agent.md`.
The prior slices are
[`v0.4-recipe-agent-live.md`](./v0.4-recipe-agent-live.md) (conversation-complete)
and the tool-driven refactor in
[`v0.4.1-recipe-agent-tool-driven.md`](./v0.4.1-recipe-agent-tool-driven.md).
Runtime-wide concepts (AgentOS, Models, Postgres + pgvector) are defined in [`Agents.md`](../../Agents.md).

**Goal:** Make the v0.4.1 recipe agent **demonstrable in the Spectres-Web chat
interface** while keeping this repository strictly backend-focused. All
frontend work — the AgentUI scaffold, the default endpoint patch, the README —
lives in `Spectres-Web`. This repository delivers the Runtime capabilities that
the web UI depends on: CORS, Markdown-formatted replies, and a one-command
local-dev orchestration script.

This is the first version where the recipe agent is **"barely usable"** through
a real UI: it looks like a product in a browser, not a curl script. The Runtime
itself stays clean of frontend code, build tooling, and Node dependencies.

Where v0.4.1 is *"tool-driven and conversation-complete but UI-empty"*, v0.5 is
**"UI-complete but personalization-empty"**: the agent answers recipe
questions in the Spectres-Web chat interface, keeps multi-turn context, and
formats replies as readable Markdown — but it still has **no saved meal plans**,
**no household profile**, and **no user memory**. Those remain deferred (see
Out of Scope).

**Definition of Done (Runtime side):**

- `Spectres-Web` is expected as a sibling directory (`../Spectres-Web`) and the
  one-command dev script knows how to launch it (§1).
- The Runtime backend exposes the AgentOS REST surface with CORS configured for
  the dev UI origin (§2).
- The recipe agent emits **formatted Markdown** (headings, lists, tables) that
  renders cleanly in AgentUI (§3).
- The full path is verified end-to-end through Spectres-Web: a user asks
  「番茄炒蛋怎么做」, gets a grounded Markdown answer, then asks
  「那它要炒几分钟？」 and the follow-up resolves using session history
  (§4).
- Telemetry stays off end to end; no anonymous run data leaves the process.
- Gates stay green on the **two-tier split** from v0.3 §10 / v0.4 §7 (§5):
  unit tests assert agent shape and CORS wiring with no live model call
  (coverage ≥ 80%); the live UI walkthrough is a manual/integration check,
  excluded from default CI.

## 1. One-Command Local Dev Launch

A demo user should not need four terminal windows. Provide a single script in
**this** repository (`Spectres-Runtime`) that brings up Postgres, the Runtime
backend, and the Spectres-Web frontend.

- **Location:** `scripts/dev.sh`.
- **Assumption:** `Spectres-Web` is checked out as a sibling directory
  (`../Spectres-Web`). If it is missing, the script exits with a clear hint
  telling the user where to clone it.
- **Steps:**
  1. `docker compose --env-file .env -f docker/compose.yaml up -d`
     (ensure Postgres is healthy via `pg_isready` poll).
  2. `uv run recipe-ingest` if the recipes table is empty (detect via a
     lightweight SQL probe; this step is **best-effort** and skipped if
     `EMBEDDER_API_KEY` is missing, so the script does not fail on a partial
     env).
  3. Start `uv run uvicorn spectres_runtime.app:app_factory --factory --reload --port 8000`
     in the background.
  4. `cd ../Spectres-Web && pnpm install && pnpm dev` in the foreground.
  5. On `Ctrl-C`, shut down the background uvicorn and optionally the Postgres
     container; document the chosen behavior.
- The script is **idempotent**: re-running it picks up the existing Postgres
  volume and existing `node_modules`.
- Document the script in `docs/environment.md` as the recommended day-one flow
  and note the sibling-directory convention for `Spectres-Web`.

**Deliverables:**

- [ ] `scripts/dev.sh` launches database → (optional ingest) → backend →
  sibling Spectres-Web frontend with one command.
- [ ] `docs/environment.md` updated with the v0.5 quick-start path, including
  the `Spectres-Web` clone location.

## 2. Backend: CORS & Dev-Facing Surface

Spectres-Web runs on `localhost:3000` and calls the AgentOS HTTP API from the
browser. The Runtime must allow that origin.

- **CORS middleware.** Mount `CORSMiddleware` on the FastAPI app for the dev
  origin only. Keep it **config-driven** (env var `AGENT_UI_ORIGIN`, default
  `http://localhost:3000`) so the same binary can be pointed at a deployed UI
  later without a code change. Do **not** use `allow_origins=["*"]` outside of
  a deliberate local-dev override.
- **Port alignment.** `app.py:main()` already binds `127.0.0.1:8000`;
  Spectres-Web's default endpoint should match. Document that `7777` referenced
  in some Agno docs is only an example.
- **No auth in v0.5.** `authorization` stays off, as in v0.4. The dev server
  remains loopback-only; CORS is the only new exposure.
- **Health / discovery endpoints remain unchanged.** `GET /health`, `GET /agents`,
  `POST /agents/recipe/runs`, and the AgentOS sessions/knowledge surface are
  provided for free by AgentOS; we write no custom endpoints.

**Deliverables:**

- [ ] `AGENT_UI_ORIGIN` wired through `Settings` + `.env.example`; CORS mounted
  on the FastAPI app in `build_app()` (production-origins list supports a
  single configured origin).
- [ ] Unit test confirms preflight `OPTIONS` from the configured origin succeeds
  and an unconfigured origin is rejected.

## 3. Agent Output Formatting (Markdown)

The v0.4.1 agent answers correctly with tool-driven retrieval but may return
dense plain text. In a chat UI, formatting is usability. Enable Markdown output
and tune the embedded instructions so replies render as clean cards in AgentUI.

- **`markdown=True` on the Agent.** Add this to `build_recipe_agent()`. Agno
  then emits content as Markdown and AgentUI renders it accordingly.
- **Instruction update.** Extend the embedded `RECIPE_AGENT_INSTRUCTIONS` (in
  `agent.py`) to ask the model for a consistent, UI-friendly shape when it
  answers from a recipe:
  - Dish name as a heading (`## 菜名`).
  - Ingredients as a bulleted list or small table.
  - Steps as a numbered list.
  - Optional notes (difficulty, time) in a short preamble.
  - Citations remain: name the dish/dishes the answer is grounded in.
- **No structured-output schema in v0.5.** Markdown is enough for "barely
  usable"; a typed `RecipeCard` response with custom frontend rendering is
  deferred.
- **Image references.** Recipe metadata carries `images`. The agent is instructed
  to **mention** images if relevant (e.g. "这道菜有配图，可在原料列表上方查看")
  but not to fabricate URLs. Actually serving images from the vendored snapshot
  is deferred (no static file host yet).

**Deliverables:**

- [ ] `build_recipe_agent()` sets `markdown=True`.
- [ ] Default instructions updated to request Markdown headings, ingredient
  list/table, numbered steps, and dish citation.
- [ ] Unit test asserts `markdown=True` and the expected instruction substring
  are present on the built agent.

## 4. End-to-End UI Verification

Prove the loop by actually using Spectres-Web against this Runtime. The
frontend-side verification steps are fully documented in
`Spectres-Web/docs/plans/v0.5-agentui-recipe-agent.md`; this section only
records the Runtime-side prerequisites and acceptance.

- **Prerequisites:** `.env` filled; database up; corpus ingested (`recipe-ingest`
  run at least once); `./scripts/dev.sh` started; `Spectres-Web` running on
  `http://localhost:3000`.
- **Acceptance (same as frontend plan):**
  1. Open `http://localhost:3000`.
  2. Confirm the endpoint sidebar shows `http://localhost:8000`; select the
     Recipe Agent.
  3. Send: 「番茄炒蛋怎么做」.
  4. Expect: the agent calls `get_recipe_detail`, then returns a grounded
     Markdown answer with a heading for 西红柿炒鸡蛋, ingredient bullets,
     numbered steps, and a citation naming the dish.
  5. Send a follow-up: 「那它要炒几分钟？」.
  6. Expect: the answer uses the prior turn and references the same dish.
  7. Refresh the browser and confirm the same `session_id` still contains the
     prior turns.
- This verification is **manual** for v0.5. Mark the deliverable complete after
  walking through the checklist.

**Deliverables:**

- [ ] Manual walkthrough checklist completed: single-turn grounded answer,
  follow-up with context, session survives refresh.

## 5. Testing & Quality

Keep the two-tier split. The UI itself is not unit-tested (it lives in another
repo and is Agno's code), but the wiring that enables it is tested here.

- **Unit tier (default CI, pure).** No DB, no network, no live model.
  - Agent shape: `markdown=True`, instructions contain Markdown cues,
    `telemetry=False`, tools (`search_recipes`, `get_recipe_detail`) registered,
    `db` injected.
  - CORS: `TestClient` `OPTIONS` preflight from `AGENT_UI_ORIGIN` succeeds;
    preflight from a different origin fails.
  - `build_app()` accepts CORS origin as an injected parameter or reads it from
    settings; tests use a test double rather than env.
  - Coverage ≥ 80% on unit-testable code.
- **Integration tier (`@pytest.mark.integration`, opt-in).**
  - Optional: a headless HTTP check that `POST /agents/recipe/runs` triggers a
    `get_recipe_detail` tool call and returns a streaming response containing
    Markdown formatting markers (`##`, `1.`).
  - Otherwise the §4 manual walkthrough satisfies the integration bar for v0.5.
- **Lint/type gate.** `ruff` / `mypy --strict` remain green. Because the
  frontend lives in a separate repo, there are no new Node/Python cross-tooling
  exclusions to maintain.

**Deliverables:**

- [ ] Unit tests cover `markdown=True`, instruction content, and CORS preflight
  behavior; coverage ≥ 80%.
- [ ] Manual or integration verification confirms Markdown-grounded answers
  through Spectres-Web.

## Explicitly Out of Scope for v0.5

v0.5 is the **"first usable demo in a web UI"** slice only.

- **Frontend code in this repository.** All UI code, Node dependencies, and
  build tooling live in `Spectres-Web`.
- **Custom frontend widgets / React components.** No recipe-card component,
  no image carousel, no step-by-step checklist widget. AgentUI is used stock
  in `Spectres-Web`.
- **Image serving.** The vendored `datasets/howtocook/dishes/` images are not
  exposed over HTTP; the agent may mention them in text but does not render them.
- **Stateful meal-plan journeys** (design §6) — saved, editable plan object and
  conversational plan adjustment. → deferred beyond v0.5.
- **User Memory / preference learning** ("likes spicy", "found it too sweet").
  → deferred beyond v0.5.
- **All Profile Management work** (`User` / `Household`, merged constraints,
  `dependencies` injection, dietary tools). This is a separate module, deferred
  per project requirement.
- **Auth / JWT / RBAC** (`authorization=True`), **os.agno.com control-plane**
  linkage, **Teams / Workflows**, **AgentFactory** / directory-driven
  construction, **Alembic migrations**, **containerization / CD**.

## Resolved Questions

- **Frontend split: separate repo.** `Spectres-Web` is a new repository; the
  Runtime stays backend-only. This avoids polluting the Python project with
  Node tooling and lets each repo evolve independently.
- **Frontend choice: Agno AgentUI.** It is the official, open-source,
  self-hosted chat interface for AgentOS. One `npx create-agent-ui@latest`
  scaffold gives a working Next.js app; no custom React work is required for
  v0.5.
- **Protocol: AgentOS REST, not AG-UI.** AgentUI calls plain AgentOS endpoints
  (`GET /agents`, `POST /agents/{id}/runs`). The separate `AGUI` interface and
  AG-UI protocol are not needed for v0.5.
- **Backend port: 8000.** Aligns with the existing `app.py:main()` dev server.
  Spectres-Web's default endpoint is patched to match; documentation clarifies
  that `7777` in upstream examples is only illustrative.
- **CORS: config-driven, single origin.** `AGENT_UI_ORIGIN` defaults to
  `http://localhost:3000`. This keeps the dev server usable while avoiding a
  wildcard in any deployed configuration.
- **Formatting: Markdown only.** A typed `RecipeCard` schema + custom widget
  would look better but is overkill for "barely usable". `markdown=True` plus
  instruction-tuned headings/lists is the right v0.5 trade-off.
- **Images: text references only.** Serving images requires a static-file route
  and potential path mapping; deferred. Recipe metadata carries `images` for
  when the frontend slice arrives.
- **Architecture: tool-driven retrieval.** v0.4.1 replaced the knowledge-based
  RAG with two explicit tools (`search_recipes` and `get_recipe_detail`). The
  agent no longer mounts a `Knowledge` object; it calls tools on demand. This
  keeps input tokens low (~1,500 vs ~60,000) and makes the agent's reasoning
  transparent.

## Open Questions

- **Should `scripts/dev.sh` leave Postgres running after `Ctrl-C`?** Starting
  cold is slow because of `pg_isready` polling; leaving it up improves iteration
  time but may surprise users who expect a clean shutdown. Decide after using
  the script for a day and document the chosen behavior.
- **Package manager in `Spectres-Web`.** `create-agent-ui` defaults to `pnpm`.
  If the team standardizes on `npm`, migrate early; otherwise keep `pnpm`.
- **Default instructions for follow-up formatting.** When the user asks a
  follow-up (e.g. 「那它要炒几分钟？」), the model may repeat the full recipe
  or answer tersely. Tune the instruction wording during §4 so the UI feels
  coherent without being verbose.
