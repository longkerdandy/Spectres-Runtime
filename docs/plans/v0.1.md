# v0.1 ‚Äî Development Environment Setup

**Goal:** Establish the engineering baseline for the Spectres-Runtime repository so that all subsequent feature work happens in a uniform, automated, reproducible environment. This release intentionally ships **no business functionality** (no AgentOS code, no agents, no database) ‚Äî only an "empty but engineering-complete" project skeleton.

**Definition of Done:**

- Any contributor can `git clone` the repo and, by following `README.md`, bring up a runnable empty application and pass all checks (`lint` / `type` / `test`) within 10 minutes.
- On every branch push and pull request, GitHub Actions runs the same checks, and code-scanning results appear in the GitHub Security tab.
- Every commit conforms to Conventional Commits; every branch and release action follows Git Flow.

**Status: ‚úÖ Completed.**

- **Package manager:** [`uv`](https://docs.astral.sh/uv/) (v0.11.17+) ‚Äî fast, native `pyproject.toml` + lockfile support, compatible with the AgentOS ecosystem.
  - Project bootstrapped with `uv init --lib --build-backend hatch` (official template, `src/` layout, `py.typed` marker included).
  - Artifacts committed: `pyproject.toml`, `uv.lock`, `.python-version`.
- **Build backend:** `hatchling` (chosen by `uv init`'s `hatch` backend option; widely used, low-friction).
- **License declared via PEP 639 SPDX:** `license = "Apache-2.0"` (the `LICENSE` file itself ships in ¬ß10).
- **Runtime dependencies (`[project.dependencies]`) ‚Äî placeholders, no wiring yet:**
  - `agno>=2.6.9` ‚Äî will host AgentOS.
  - `fastapi>=0.136.3` ‚Äî web entry point.
  - `uvicorn[standard]>=0.48.0` ‚Äî ASGI server.
- **Dependency groups (PEP 735 `[dependency-groups]`):**
  - `lint` ‚Äî `ruff` only. Kept minimal so the CI lint job stays sub-second.
  - `test` ‚Äî `pytest`, `pytest-asyncio`, `pytest-cov`.
  - `dev` ‚Äî `mypy`, `pre-commit`, plus `include-group = "lint"` + `include-group = "test"`. This is uv's default group, so `uv sync` (no flags) installs the full developer environment in one command.
- **Why three groups, not one:** lets CI install only what each job needs (`uv sync --only-group lint` for the lint job, `--only-group test` for tests), keeps production images slim (`uv sync --no-dev`), and gives each group a clear semantic role. See `docs/plans/v0.1.md` design discussion.
- **Why `ruff` alone in `lint`:** ruff already subsumes flake8 + isort + black + pyupgrade + bugbear + simplify + a baseline of bandit-style security rules (`S`). Deep security scanning is handled by CodeQL (¬ß8); CVE scanning by Dependabot (¬ß8). Additional lint tools (codespell / vulture / interrogate) can be added incrementally if a concrete need appears.
- **Standard developer commands (to be documented in `README.md` under ¬ß10):**

  ```bash
  uv python install 3.12        # one-time, if uv hasn't already
  uv sync                       # installs runtime + dev (= lint + test + mypy + pre-commit)
  uv run python -c "import spectres_runtime; print(spectres_runtime.hello())"
  ```

- **CI-oriented commands (to be wired in ¬ß7):**

  ```bash
  uv sync --frozen --only-group lint     # lint job
  uv sync --frozen --only-group test     # test job
  uv sync --frozen --no-dev              # production / container image
  ```

## 2. Empty Project Skeleton

**Status: ‚úÖ Completed.**

Minimal runnable structure, to be extended by later releases:

```
src/
‚îî‚îÄ‚îÄ spectres_runtime/
    ‚îú‚îÄ‚îÄ __init__.py     # package metadata (__version__)
    ‚îú‚îÄ‚îÄ py.typed        # PEP 561 marker (shipped by uv init --lib)
    ‚îî‚îÄ‚îÄ app.py          # FastAPI() instance + /healthz endpoint + main()
tests/
‚îî‚îÄ‚îÄ test_healthz.py     # smoke test via fastapi.testclient.TestClient
```

- **Layout:** `src/` layout (from `uv init --lib`), prevents implicit imports and forces installation through the build backend.
- **FastAPI app:** module-level `app: Final[FastAPI]` so `uvicorn spectres_runtime.app:app` works directly. Title/version/description set from project metadata.
- **`/healthz` endpoint:** async, no I/O, returns `{"status": "ok"}`. Serves as both a skeleton self-check and a probe for CI / container health checks. Tagged `meta` so it's grouped separately from future business endpoints in the OpenAPI schema.
- **`main()` callable:** kept in `app.py` for a future `[project.scripts]` entry point. **Not** registered in `pyproject.toml` for v0.1 ‚Äî `uvicorn spectres_runtime.app:app` is the standard launch command; scripts entry can be added later without breaking anything.
- **Smoke test:** uses `fastapi.testclient.TestClient` (synchronous wrapper, no event loop boilerplate). One test asserts status `200` and exact JSON payload. Verified passing with `uv run pytest`.
- **Known warning (non-blocking):** Starlette's `TestClient` emits `StarletteDeprecationWarning: Using httpx with starlette.testclient is deprecated; install httpx2 instead`. Test still passes; `httpx2` is too new to adopt. Suppressed in ¬ß3 via pytest `filterwarnings`.
- **Launch verified:** `uv run uvicorn spectres_runtime.app:app` boots successfully; `GET /healthz` returns `{"status": "ok"}`; `/openapi.json` auto-generated with the single endpoint listed.

## 3. Code Quality Tool Configuration

**Status: ‚úÖ Completed.**

All configuration lives in `pyproject.toml` under `[tool.*]`; no scattered `setup.cfg` / `.flake8` / `mypy.ini`. All three tools verified passing after install (`uv run ruff check . && uv run ruff format --check . && uv run mypy && uv run pytest`).

- **Ruff** (`[tool.ruff]` + `[tool.ruff.lint]` + `[tool.ruff.format]`)
  - `line-length = 100`, `target-version = "py312"`.
  - Rule sets enabled: `E, W, F, I, B, UP, SIM, RUF` ‚Äî the consensus modern-Python baseline used by FastAPI / httpx / pydantic. Kept intentionally small; expand only when a concrete need appears.
  - `per-file-ignores`: `tests/**/*.py` is allowed `S101` (`assert`) since pytest's model is built on it.
  - Formatter aligned with Black defaults: double quotes, spaces, LF ‚Äî explicit choices keep cross-platform diffs stable.
- **Mypy** (`[tool.mypy]`)
  - `python_version = "3.12"`, `strict = true` (bundles ~10 individual checks).
  - `files = ["src", "tests"]` ‚Äî type-checks both production code and tests.
  - `show_error_codes = true`, `pretty = true` ‚Äî error codes enable surgical `# type: ignore[code]` instead of blanket ignores.
  - Extra warn flags on: `warn_unused_ignores`, `warn_redundant_casts`, `warn_return_any`.
  - Override for `tests.*`: `disallow_untyped_decorators = false` so untyped third-party pytest decorators don't require stubs.
- **Pytest** (`[tool.pytest.ini_options]`)
  - `minversion = "9.0"`, `testpaths = ["tests"]`.
  - `addopts`: `-ra`, `--strict-markers`, `--strict-config`, `--cov=spectres_runtime`, `--cov-report=term-missing`, `--cov-fail-under=80`.
  - `filterwarnings`: suppresses Starlette's environmental `StarletteDeprecationWarning` (httpx ‚Üí httpx2 nudge).
  - `asyncio_mode = "auto"` ‚Äî `async def test_*` works without per-test markers.
- **Coverage.py** (`[tool.coverage.run]` + `[tool.coverage.report]`)
  - `source = ["src/spectres_runtime"]`, `branch = true` ‚Äî both edges of conditionals must be exercised.
  - `exclude_lines`: `pragma: no cover`, `if __name__ == "__main__":`, `raise NotImplementedError`.
  - `show_missing = true`, `skip_covered = false` so all files appear in the report.
- **Coverage gate met:** baseline 100% (`__init__.py` + `app.py`); `main()` excluded via `# pragma: no cover` as a thin uvicorn wrapper exercised manually, not in unit tests.
- **Comment hygiene:** `pyproject.toml` comments explain *why* and *what* (decisions, semantics); status-shaped comments (`yet` / `for now` / `will be` / `once`) are avoided to prevent rot as the project evolves.

## 4. Pre-commit Hooks

**Status: ‚úÖ Completed.**

- **File:** `.pre-commit-config.yaml` at the repo root. All hook repos are pinned to specific `rev:` tags so behaviour is reproducible across clones and time. Bump with `uv run pre-commit autoupdate`.
- **Hooks installed** (in stage order):
  - **`pre-commit/pre-commit-hooks` v6.0.0** ‚Äî file hygiene: `check-merge-conflict`, `end-of-file-fixer`, `trailing-whitespace`, `check-yaml`, `check-toml`, `check-added-large-files` (1 MB cap).
  - **`astral-sh/ruff-pre-commit` v0.15.15** ‚Äî `ruff-check --fix` then `ruff-format`. Same Ruff version as the `lint` dependency group, so local and hook behaviour can't drift.
  - **`pre-commit/mirrors-mypy` v2.1.0** ‚Äî strict type check, scoped to `^src/` for snappy commits. Carries `additional_dependencies` for `fastapi`, `uvicorn[standard]`, and `agno` because each pre-commit hook runs in its own isolated venv and can't see the project `.venv`. CI runs the full mypy invocation (src + tests).
  - **`compilerla/conventional-pre-commit` v4.3.0** ‚Äî runs on the `commit-msg` stage; rejects commit messages that don't follow Conventional Commits.
- **Installation (one-time per clone):**

  ```bash
  uv run pre-commit install                          # .git/hooks/pre-commit
  uv run pre-commit install --hook-type commit-msg   # .git/hooks/commit-msg
  ```

- **Manual full-repo run:** `uv run pre-commit run --all-files` ‚Äî all hooks pass on the current skeleton.
- **Commit-msg gate verified:** ad-hoc tests with a bad message ("added stuff") fail; a Conventional Commits message ("chore: add pre-commit config") passes.
- **CI vs. pre-commit boundary:** the two run the same checks but along different paths ‚Äî pre-commit uses isolated per-hook venvs for speed and runs on staged files, CI uses the project `.venv` and runs on the full tree. This is intentional; see ¬ß7 for the CI side.

## 5. Commit Message Conventions

**Status: ‚úÖ Completed.**

- **Convention:** [Conventional Commits 1.0](https://www.conventionalcommits.org/).
  - Allowed `type` values: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`.
  - `scope` is optional; lowercase module / area names recommended (e.g. `agent`, `memory`, `ci`, `deps`).
  - `subject` style: imperative mood, lowercase, no trailing period, ‚â§ ~50 chars.
- **Single source of truth: `CONTRIBUTING.md` at the repo root.** Houses both the machine-enforced rules and the project conventions, split into two clearly-labelled subsections so contributors know which lines a tool will actually reject:
  - **Machine-enforced** (what the `commit-msg` hook from ¬ß4 validates): type whitelist, `type(scope): subject` shape, `!` breaking-change marker, non-empty subject, blank line before body.
  - **Project conventions** (NOT machine-enforced, kept as a written agreement for humans and AI coding assistants): lowercase subject, ‚â§ 50 chars, imperative mood, no trailing period, scope naming, ~72-char body wrap, footer keywords (`BREAKING CHANGE`, `Refs`, `Closes`).
- **`Agents.md` carries a pointer, not the rules.** A short `## Repository Conventions` section at the bottom of `Agents.md` instructs AI coding assistants (which auto-load `Agents.md` into their system prompt) to read `CONTRIBUTING.md` before making changes. This keeps `Agents.md` focused on *what the system builds* and avoids the "Coding Agents vs. Agno `Agent`" naming collision that arose from earlier drafts.
- **No `.gitmessage` template.** An editor-pre-fill template was prototyped and removed: AI agents commit via `git commit -m`, which bypasses `commit.template` entirely, and the project is single-developer with no editor-driven commit workflow. Keeping a template would create a second source of truth that drifts from `CONTRIBUTING.md` without adding enforcement. The previous `git config --local commit.template` is unset.
- **Enforcement layers:**
  - **Local:** `commit-msg` pre-commit hook (`compilerla/conventional-pre-commit`, configured in ¬ß4) rejects structurally non-conforming messages.
  - **CI:** PR title validation via `amannn/action-semantic-pull-request` (planned in ¬ß7).
  - **None of the above enforces the "Project conventions" tier** ‚Äî that gap is acknowledged in `Agents.md` and accepted for v0.1.
- **README quick-start** covers ¬ß1‚Äì¬ß5 commands in one block: `uv python install 3.12`, `uv sync`, `pre-commit install` (√ó2 stages), then smoke tests. Full README polish is deferred to ¬ß10.

## 6. Git Flow Convention

Classic Git Flow, simplified ‚Äî documented in `CONTRIBUTING.md` (created alongside this release):

| Branch | Purpose | From | Merges into |
|---|---|---|---|
| `main` | Released, stable, tagged | ‚Äî | ‚Äî |
| `develop` | Integration branch; the working surface of the next release | `main` | `main` (via release) |
| `feature/<name>` | New feature | `develop` | `develop` |
| `fix/<name>` | Non-urgent bug fix | `develop` | `develop` |
| `release/<version>` | Release prep ‚Äî version bump / changelog / minor fixes only | `develop` | `main` + `develop` |
| `hotfix/<name>` | Urgent production fix | `main` | `main` + `develop` |

- **Protection rules (GitHub Settings, documented):** `main` and `develop` reject direct pushes; PR + ‚â• 1 review + all-green CI required.
- **Tags:** semantic versioning `vMAJOR.MINOR.PATCH`. This release ships as `v0.1.0`.
