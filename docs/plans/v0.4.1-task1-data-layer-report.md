## 结论

### 1. `recipes` 表 Schema

`public.recipes` 表由 Agno 的 `PgVector` + `Knowledge.insert()` 自动创建，列结构如下：

| 列名 | 类型 | 说明 | 来源 |
|---|---|---|---|
| `id` | `SERIAL PRIMARY KEY` | 自增主键 | PgVector 默认 |
| `name` | `VARCHAR` | **recipe_id**（如 `howtocook/aquatic/咖喱炒蟹`） | `sink.py:65` |
| `content` | `TEXT` | **完整 Markdown 菜谱**（被嵌入的文本） | `sink.py:66` → Agno stores as `content` |
| `embedding` | `USER-DEFINED` | 嵌入向量 (1024-dim) | PgVector 默认 |
| `meta_data` | `JSONB` | 结构化 metadata | `sink.py:67, 88-96` → Agno stores as `meta_data` |
| `content_hash` | `VARCHAR` | 内容哈希（Agno 内部使用） | PgVector 默认 |
| `created_at` / `updated_at` | `TIMESTAMP` | 时间戳 | PgVector 默认 |

**关键确认：**
- `name` 列存储的是 `recipe.id`，格式为 `howtocook/<category>/<dish_name>`
- `content` 列存储的是完整的 Markdown 菜谱（含步骤、用量、小贴士）
- `get_recipe_detail` 应该直接 `SELECT content FROM recipes WHERE name = %s`

### 2. Metadata JSONB 结构

由 `sink.py:_metadata()` 方法生成，结构确定：

```json
{
  "recipe_id": "howtocook/aquatic/咖喱炒蟹",
  "name": "咖喱炒蟹",
  "source": "howtocook",
  "ref": "aquatic/咖喱炒蟹.md",
  "category": ["aquatic"],
  "difficulty": 4,
  "images": [],
  "ingredients": [
    {"name": "青蟹", "optional": false},
    {"name": "咖喱块", "optional": false}
  ]
}
```

**关键确认：**
- `category` 是字符串数组（单元素），如 `["aquatic"]`
- `ingredients` 是对象数组，每个对象有 `name` 和 `optional` 字段
- **没有 `description` 字段** —— `Recipe.description` 存在但未被写入 metadata（`sink.py:88-96` 未包含 `description`）

### 3. Category 取值集合

来自 `datasets/howtocook/dishes/` 的子目录名：

```
aquatic        (~30 recipes)   水产
breakfast      (~15 recipes)   早餐
condiment      (~20 recipes)   酱料/调味品
dessert        (~25 recipes)   甜品
drink          (~15 recipes)   饮品
meat_dish      (~80 recipes)   荤菜
semi-finished  (~10 recipes)   半成品
soup           (~40 recipes)   汤
staple         (~50 recipes)   主食
template       (~5 recipes)    模板（非菜谱）
vegetable_dish (~60 recipes)   素菜
```

**总计：~360 recipes**（与之前调研的 357 一致）

**关键确认：**
- `template` 不是真实菜谱，应该被 `search_recipes` 自动排除
- 其他 10 个 category 是有效的菜谱分类

### 4. `agno_knowledge` 表

由 Agno 的 `contents_db`（`PostgresDb`）管理，存储内容追踪信息。`Knowledge.insert()` 会同时写入：
- `recipes` 表（PgVector）：embedding + content + meta_data
- `agno_knowledge` 表（PostgresDb）：内容追踪（name, hash, 等）

**结论：** `get_recipe_detail` **不需要**查 `agno_knowledge` 表。`recipes` 表的 `content` 列已经有完整内容。

### 5. Content 长度估算

基于 `datasets/howtocook/` 目录下的 Markdown 文件大小估算：
- 最短菜谱（如凉拌菜）：~500 字符
- 最长菜谱（如小龙虾）：~3,000+ 字符
- **中位数：~1,500 字符**

转换为 token（中文 ≈ 1.5 token/字）：
- 中位数：~1,000-1,500 tokens
- 最大值：~2,000-3,000 tokens

### 6. Ingredient 名称特点

从 `catalog/recipes.jsonl` 抽样分析：
- 使用**中文全称**，如"西红柿"而非"番茄"，"洋芋"而非"土豆"
- 调料名称具体，如"郫县豆瓣"、"黄豆酱"、"青花椒"
- 有复合食材，如"葱姜蒜"（视为一个 ingredient）
- 精确匹配会有**同义词问题**（番茄 vs 西红柿），v0.4.1 接受此限制

## 对后续开发的影响

### search_recipes 的 SQL 设计

基于以上确认，SQL 应如下：

```sql
SELECT
    name AS recipe_id,
    meta_data->>'name' AS name,
    meta_data->'category' AS category,
    meta_data->>'difficulty' AS difficulty,
    meta_data->>'name' AS description,  -- 暂无 description，fallback 到 name
    meta_data->'ingredients' AS ingredients,
    embedding <=> %s::vector AS distance
FROM recipes
WHERE 1=1
  AND NOT meta_data->'category' @> '["template"]'
  -- category filter
  AND meta_data->'category' @> %s::jsonb
  -- must_include filter
  AND meta_data->'ingredients' @> %s::jsonb
  -- must_exclude filter
  AND NOT meta_data->'ingredients' @> %s::jsonb
ORDER BY embedding <=> %s::vector
LIMIT %s
```

### get_recipe_detail 的 SQL 设计

```sql
SELECT content
FROM recipes
WHERE name = %s
LIMIT 1
```

### 需要额外处理的问题

1. **`meta_data` 中没有 `description`**
   - 短期：用 `name` 或运行时从 `content` 截断前 120 字作为 description
   - 长期（v0.4.2+）：在 ingestion 时预生成 `description` 存入 meta_data

2. **PgVector 列名确认** ✅
   - 已确认：`content`（不是 `text_content`），`meta_data`（不是 `metadata`），`embedding`（不是 `vector`）

3. **`name` 列的唯一性**
   - 由 `upsert=True` 保证：`name` 是稳定 key
   - `get_recipe_detail` 用 `WHERE name = %s` 是安全的
