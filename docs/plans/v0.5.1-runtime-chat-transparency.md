# v0.5.1 — Spectres-Runtime: Chat Transparency

**Status:** planned.

**Goal:** Feed the Spectres-Web v0.5.1 "Chat Transparency" slice with the
runtime data it needs: reasoning steps, tool-call details, and per-message token
usage. This slice changes no frontend code; it only adjusts the Runtime so the
existing AgentOS run stream exposes the fields AgentUI expects.

The companion frontend plan lives in the sibling repository `Spectres-Web` at
`docs/plans/v0.5.1-ui-polish.md`. The preceding Runtime slice is
[`v0.5-agentui-recipe-agent.md`](./v0.5-agentui-recipe-agent.md).

**Important prerequisite:** Task 1 inspection
([`v0.5.1-task1-stream-inspection-report.md`](./v0.5.1-task1-stream-inspection-report.md))
found that the recipe agent currently has **no tools registered**
(`agent.tools == []`) and still instructs the model to "search the recipe
knowledge base". As a result, the stream emits no `ToolCallStarted` /
`ToolCallCompleted` events. **Tool registration must be fixed before any
transparency work can be validated.**

**Definition of Done (Runtime side):**

- The recipe agent correctly registers and invokes `search_recipes` and
  `get_recipe_detail` (§1).
- Tool calls appear in the run stream as `ToolCallStarted` /
  `ToolCallCompleted` events with name, arguments, result, error flag, and
  timing (§2).
- The recipe agent emits reasoning content that AgentUI can render as
  `message.extra_data.reasoning_steps` (§3).
- Token metrics (`input_tokens` / `prompt_tokens`, `output_tokens` /
  `completion_tokens`, `time`) are present in the run response when the model
  provider returns them (§4).
- The default CORS origin and streaming endpoint contract remain unchanged.
- Unit tests assert agent shape and stream metadata; integration tests confirm
  the live stream contains the expected event types (§5).

## 1. Fix Recipe Agent Tool Registration (Blocker)

The v0.4.1 refactor intended to make the recipe agent tool-driven, but the
running agent has `agent.tools == []`. The model therefore hallucinates
retrieval instead of calling `search_recipes` / `get_recipe_detail`. No
transparency feature can be tested until this is fixed.

### 1.1 Wire the tools into `build_recipe_agent()`

In `src/spectres_runtime/recipe_agent/agent.py`:

```python
from spectres_runtime.recipe_agent.tools import (
    build_get_recipe_detail_tool,
    build_search_recipes_tool,
)

def build_recipe_agent(
    settings: Settings,
    *,
    model: Model | None = None,
    db: BaseDb | None = None,
) -> Agent:
    ...
    return Agent(
        id=RECIPE_AGENT_ID,
        name=RECIPE_AGENT_NAME,
        model=model or settings.recipe_agent.build_chat_model(),
        db=db or build_db(settings),
        tools=[
            build_search_recipes_tool(settings),
            build_get_recipe_detail_tool(settings),
        ],
        instructions=instructions,
        add_history_to_context=True,
        num_history_runs=settings.recipe_agent.num_history_runs,
        user_id=DEFAULT_USER_ID,
        telemetry=False,
        markdown=True,
    )
```

### 1.2 Update the embedded instructions and tool docstrings

The current instructions say "Search the recipe knowledge base". Change them
to instruct the agent to **call the registered tools**:

- For discovery / menu planning: call `search_recipes`.
- For detailed steps: call `get_recipe_detail` with the `recipe_id` from
  `search_recipes`.
- Ground every answer in the tool results; do not invent recipes.
- Keep the existing Markdown formatting requirements (headings, lists, tables)
  and dish-citation rules.

**Important:** Agno automatically appends each tool's docstring to the system
instructions sent to the model. Therefore the agent instructions and the tool
docstrings must be coherent:

- Remove any mention of "knowledge base" from both the agent instructions and
  the tool docstrings.
- Use consistent terminology across both (e.g., "recipe catalog", "recipe_id",
  "search", "retrieve").
- Ensure the tool docstrings state **when** to call the tool and **what** it
  returns, so the model can choose correctly without contradicting the agent's
  high-level instructions.

### 1.3 Verify tool registration

Directly inspect the built agent:

```python
agent = build_recipe_agent(settings)
assert len(agent.tools) == 2
assert any(t.name == "search_recipes" for t in agent.tools)
assert any(t.name == "get_recipe_detail" for t in agent.tools)
```

**Deliverables:**

- [x] `build_search_recipes_tool()` and `build_get_recipe_detail_tool()` are
  imported and registered in `build_recipe_agent()`.
- [x] Default instructions updated to reference tools, not the knowledge base.
- [x] Tool docstrings updated to align with the agent instructions.
- [x] Unit tests assert the agent exposes `search_recipes` and

## 3. Tool-Call Visibility

With tools registered, AgentOS should wrap calls in `ToolCallStarted` /
`ToolCallCompleted` events. This section verifies the fields and fixes any
remaining gaps.

### 3.1 Confirm default behavior

With the recipe agent registered in AgentOS and `stream=true`, ask:

> "Plan a 4-dish + 1-soup menu."

Expect the SSE stream to include:

- `ToolCallStarted` with `tool_name: "search_recipes"` and `tool_args` showing
  the JSON arguments.
- `ToolCallCompleted` with the JSON result and `metrics.time`.
- No custom middleware should be required for basic visibility.

### 3.2 Handle missing fields

If the default stream omits any of the fields the Web plan needs (e.g., timing,
formatted arguments, or error flag), add a thin wrapper or hook rather than
rewriting the tool:

- Prefer Agno's `pre_tool_call` / `post_tool_call` hooks if they can enrich the
  event payload.
- Only introduce a custom endpoint if AgentOS events are fundamentally
  insufficient; custom endpoints break the stock AgentUI integration.

### 3.3 Keep tools fast and observable

The tool implementations in `src/spectres_runtime/recipe_agent/tools.py` already
return JSON strings. Ensure errors are raised or returned in a shape that Agno
marks as `tool_call_error`. For example, if `search_recipes` receives an invalid
category, the current code returns a JSON `{"error": ...}` — decide whether this
should be a tool-level error event or a normal result. The UI will highlight
`tool_call_error=true`; normal error payloads may be missed.

**Deliverables:**

- [ ] Verified that `search_recipes` and `get_recipe_detail` appear as distinct
  tool-call events in the SSE stream.
- [ ] Tool-call events include `tool_name`, `tool_args`, result content, and
  timing.
- [ ] Error behavior aligned with UI expectations (tool-call error flag or
  explicit payload).

## 4. Reasoning Transparency

The recipe agent currently does **not** enable reasoning explicitly. The model
used in Task 1 happened to emit reasoning tokens, which Agno surfaced as a flat
`reasoning_content` string. For AgentUI to show expandable reasoning steps, the
Runtime must turn on an Agno reasoning mode that produces structured content or
explicit reasoning tool calls.

### 4.1 Choose a reasoning strategy

Agno offers two main paths:

1. **Model-native reasoning** — set `reasoning=True` on the `Agent` and use a
   model that exposes reasoning tokens (e.g., OpenAI reasoning models, Claude
   with `thinking`, Gemini with `thinking_budget`, DeepSeek reasoning variants).
   This produces `reasoning_content` string deltas, not structured steps.
2. **Tool-based reasoning** — add `ReasoningTools` to the agent's tool list. The
   model calls explicit `think()` / `analyze()` tools, which appear as normal
   tool calls and can be rendered as reasoning steps.

For v0.5.1, choose the path that matches the configured chat model. Since the
recipe agent is provider-agnostic (`OpenAILike` with a configurable base URL),
tool-based reasoning is the safer default: it works with any function-calling
model and gives the Web plan structured steps to render.

### 4.2 Add `ReasoningTools` to the recipe agent

In `src/spectres_runtime/recipe_agent/agent.py`:

```python
from agno.tools.reasoning import ReasoningTools

def build_recipe_agent(settings: Settings, ...) -> Agent:
    ...
    return Agent(
        ...,
        tools=[
            build_search_recipes_tool(settings),
            build_get_recipe_detail_tool(settings),
            ReasoningTools(add_instructions=True),
        ],
        ...,
    )
```

Update the embedded instructions to explicitly ask the agent to show its
reasoning when planning menus or comparing dishes. Keep the existing Markdown
formatting instructions intact.

### 4.3 Verify reasoning events in the stream

Run a menu-planning request with `stream=true` and confirm the stream contains
reasoning-related content that maps to the fields AgentUI expects. If Agno emits
`reasoning_content` but AgentUI expects `reasoning_steps`, document the mismatch
and decide whether to:

- Transform events in a custom hook, or
- Update the Web plan's field expectations.

**Deliverables:**

- [ ] `ReasoningTools` added to the recipe agent.
- [ ] Default instructions updated to encourage step-by-step reasoning.
- [ ] Streaming response verified to contain reasoning content.

## 5. Token Usage Metrics

Token counts and response latency come from the model provider and are surfaced
by Agno in the run response / final event. Task 1 confirmed they are already
present; this section keeps them intact after the §1–§4 changes.

### 5.1 Confirm metrics from the provider

The recipe agent uses `OpenAILike`, which wraps any OpenAI-compatible endpoint.
Most providers return `usage` in the response, but field names vary:

- `prompt_tokens` / `completion_tokens`
- `input_tokens` / `output_tokens`

AgentUI's v0.5.1 plan checks both aliases. Confirm which fields the configured
provider returns by inspecting a non-streaming `POST /agents/recipe/runs` with
`stream=false`.

### 5.2 Surface metrics in streaming mode

Streaming responses emit token usage in the final `RunCompleted` and
`ModelRequestCompleted` events. Verify that:

- The final event includes `metrics` or `usage` with token counts.
- The `time` / `duration` field is present.
- AgentUI can read these values without a custom endpoint.

### 5.3 No estimation in the Runtime

As with the Web plan, do not estimate or synthesize token counts. If the model
provider does not return usage, the UI will hide the footer. Only ensure that
whatever the provider gives us is forwarded untouched.

**Deliverables:**

- [ ] Verified token fields are still returned after tool / reasoning changes.
- [ ] Confirmed metrics appear in both streaming and non-streaming run responses.
- [ ] No token estimation logic added.

## Out of Scope for v0.5.1

- Custom frontend widgets, React components, or image rendering.
- New AgentOS endpoints beyond what Agno generates automatically.
- Auth / JWT / RBAC.
- Session management features (clear conversation, retry, copy message).
- Deployment / containerization work.
- Structured output schemas or recipe cards.
- Changing the tool semantics beyond making them callable.

## Acceptance Criteria

- `build_recipe_agent()` registers `search_recipes` and `get_recipe_detail`, and
  the agent invokes them during a real run.
- `POST /agents/recipe/runs?stream=true` emits `ToolCallStarted` and

- Token usage and timing metrics are present in run responses when the model
  provider returns them.
- Missing data degrades gracefully: AgentUI hides sections rather than crashing.
- `pytest`, `ruff`, and `mypy` pass; coverage ≥ 80%.
