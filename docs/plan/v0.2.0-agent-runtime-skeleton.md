## 2. Context

`v0.1.0` established the development environment, quality workflow, and GitHub security scanning. `v0.2.0` is the first application milestone. It focuses on **structure over behavior**: we want the right modules, interfaces, and configuration in place so that future milestones (memory, knowledge base, Client/Edge protocols) can plug in cleanly.

The design is inspired by `agentos-docker-template` but is intentionally **not a direct copy**. Spectres Runtime has its own architecture (Team Leader / Slave agents, Client API, Edge Gateway Proxy) and will evolve separately.

Key constraints:

- Use Agno's `Agent`, `AgentOS`, and `Db` abstractions where they fit.
- Session and chat history must be persistent, but memory (user profiles, long-term recall) is explicitly out of scope.
- The database choice in this milestone must support future `mem0` or `hindsight` integration without a later migration.
- Code should be minimal: stubs, type annotations, and small tests are enough. The server does not need to start successfully in CI.

## 3. Scope

### 3.1 In Scope

1. **Agno and runtime dependencies**
   - Add `agno`, `fastapi`, `openai`, `pydantic-settings`, `psycopg`, and `sqlalchemy` to `pyproject.toml`.
   - Update `uv.lock`.

2. **Project source structure**
   - Create `src/spectres/` subpackages:
     - `agents/` — Agent definitions (Team Leader Agent stub).
     - `tools/` — Built-in tool registrations.
     - `db/` — Database adapter/factory.
     - `config.py` — Typed settings via `pydantic-settings` (database URL and Team Leader LLM options).
     - `main.py` — Minimal `AgentOS` entry point stub exposing a FastAPI app.
   - Keep each module small; empty `__init__.py` files and placeholder functions are acceptable.

3. **Database environment**
   - Add a local PostgreSQL service via `docker-compose.yml`.
   - Run the database in **Docker Desktop** for local development.
   - Provide an example environment file (`.env.example`) with database connection variables and Team Leader LLM variables.
   - Add a database adapter/factory in `src/spectres/db/` that returns an Agno `PostgresDb` instance.
   - Document how to start the database in Docker Desktop for local development.

4. **Session / chat-history management**
   - Wire `PostgresDb` into the Team Leader Agent stub so sessions can be persisted.
   - Configure `add_history_to_context=True` and `num_history_runs` on the agent.
   - A dedicated `sessions/` package is **not** required; session storage is handled directly through Agno's `db` abstraction.

5. **Basic built-in tools**
   - Register **two** built-in Agno toolkits:
     - `CalculatorTools` — arithmetic helper.
     - `ShellTools` — run shell commands (e.g. `date`, `ls`).
   - Place tool wiring in `src/spectres/tools/builtin.py`.

6. **Tests**
   - Add minimal tests verifying that modules import cleanly, configuration can be loaded, the database adapter builds, tools are registered, the Team Leader Agent can be created, and the AgentOS app is a FastAPI instance.
   - Tests should not require a running database or model API keys.

7. **Documentation**
   - Update `README.md` with the new source layout and database startup instructions.

### 3.2 Out of Scope

- Long-term memory (`mem0`, `hindsight`, or Agno memory abstractions).
- Knowledge base / RAG / embeddings.
- Edge Gateway Proxy or device-control tools.
- Client API / AG-UI protocol endpoints.
- Authentication, authorization, multi-tenancy.
- Multi-agent teams, routing, or workflows.
- Docker image build for the runtime itself.
- Production deployment manifests.
- A fully runnable server in CI.

## 5. Database Selection

### 5.1 Requirements for v0.2.0

- Persistent storage for Agno **sessions** and **chat history**.
- Simple local development setup.
- Clean async/sync adapter interface for future milestones.

### 5.2 Requirements for Future Milestones

- Support for `mem0` or `hindsight` as the long-term memory backend.
- Potential support for vector storage when the knowledge base / RAG is added.
- Production-ready operation.

### 5.3 Local Development Environment

The local PostgreSQL instance will run inside **Docker Desktop** via `docker-compose.yml`. This keeps the development environment isolated from the host system and makes it easy to destroy and recreate the database during early iteration.

## 5.4 Options

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| **SQLite** (`SqliteDb`) | Zero setup, single file, works offline. | Poor concurrency, no vector extension, likely requires migration to Postgres for memory/RAG later. | **Rejected** for the main storage backend. |
| **PostgreSQL** (`PostgresDb`) | Agno's recommended production DB; `pgvector` extension available for future RAG; works with managed services (Neon, Supabase, RDS); mem0/hindsight both support Postgres. | Requires running a Postgres service locally. | **Recommended**. |
| **MongoDB** (`MongoDb`) | Flexible schema, good for JSON-heavy session data. | Not the default in Agno docs; mem0/hindsight support less certain; adds a second database family to the stack. | **Rejected**. |
| **Redis** (`RedisDb`) | Fast, cache-friendly. | Not a durable primary store for sessions; not suitable for knowledge base. | **Rejected** for primary storage. |

### 5.4 Recommendation

Use **PostgreSQL with the `pgvector` extension** as the primary database from v0.2.0 onward.

Rationale:

1. **Future-proof**: `pgvector` is already the de-facto choice for Agno knowledge-base embeddings, and both `mem0` and `hindsight` document Postgres support.
2. **Single engine**: sessions, chat history, future memory, and future knowledge vectors can live in one database.
3. **Agno alignment**: `PostgresDb` is the default and most mature storage backend in Agno.
4. **Local parity with production**: managed Postgres services (Neon, Supabase, AWS RDS) are drop-in replacements for the local Docker container.

For v0.2.0, the runtime code only needs to construct `PostgresDb` from a `DATABASE_URL`; actual tables are managed by Agno's storage layer. The Docker Compose file will use the Agno-provided `agnohq/pgvector:18` image so the `pgvector` extension is available immediately and the local environment matches the official AgentOS template.
